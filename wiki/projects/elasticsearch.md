# Elasticsearch

## What It Is

Elasticsearch is a distributed search and analytics engine built on Apache Lucene. Released by Elastic in 2010, it stores data as JSON documents across sharded indices and exposes a REST API for querying. In AI/ML contexts, it functions as the retrieval backend in RAG pipelines, a vector database for semantic search, and a persistent store for agent memory systems.

The engine has accumulated well over 70,000 GitHub stars. Adoption metrics are largely self-reported by Elastic; independent audits of query performance vary significantly by workload and cluster configuration.

## Core Mechanism

Elasticsearch indexes documents into inverted indices for keyword (BM25) search and HNSW graphs for vector (dense) search. Both can run against the same document in a single query, which is what makes it useful for hybrid retrieval.

The search pipeline for RAG workloads typically combines:

- **`MatchTextExpr`** — BM25 full-text matching across tokenized fields
- **`MatchDenseExpr`** — approximate nearest neighbor search over float32 or quantized vectors
- **Semantic text fields** — via the `semantic_text` mapping type, which embeds at index time using a configured model (Elastic's ELSER sparse encoder or a dense model)

ELSER (Elastic Learned Sparse EncodeR) deserves specific mention. Rather than producing dense vectors, it produces sparse token-weight representations that align with the inverted index structure. This lets semantic matching run on BM25 infrastructure with no separate vector index, at the cost of model inference during both indexing and querying.

Document-level security applies row-level filters at query time based on the authenticated user's role. A role definition includes a query fragment that Elasticsearch ANDs into every request. This is how the agentic memory architecture in [Elasticsearch Labs documentation](../../raw/articles/elasticsearch-labs-ai-agent-memory-agentic-ai-memory-management-with.md) isolates episodic memory by identity: innie and outie roles carry built-in `memory_type` filters, so the application never needs to handle memory scoping in code.

Multi-field mappings let a single source field serve multiple index structures simultaneously. A `memory_text` field can be indexed as both plain text (for BM25) and as a semantic text embedding (for vector search), queried in a single hybrid request.

## Role in RAG Systems

RAGFlow uses Elasticsearch as its default document engine, with [pluggable alternatives](../../raw/repos/infiniflow-ragflow.md) including Infinity, OpenSearch, and OceanBase. All backends implement the same factory interface, so chunked documents, embeddings, keyword tokens, and knowledge graph structures land in the same index regardless of backend. For GraphRAG workloads, entity relationships generated by the Leiden community detection algorithm are stored alongside chunk data in Elasticsearch indices.

The standard ingestion path for a chunk:

1. Parse → 2. Embed (dense vector) → 3. Tokenize (sparse/BM25) → 4. Index into Elasticsearch with metadata fields

At query time, multi-recall retrieval fans out across dense, sparse, and optionally graph retrieval, then fuses and re-ranks results before passing context to the LLM.

## Strengths

**Hybrid retrieval in one system.** Running BM25 and vector search against the same document store eliminates synchronization between separate keyword and vector databases. For most retrieval workloads this matters operationally more than it matters for benchmark performance.

**Metadata filtering before semantic scoring.** Structured filters (user ID, timestamp, document type, memory context) narrow the candidate set before vector scoring runs. This reduces both latency and the effective search space, which matters when memory stores grow large.

**Document-level security at the engine layer.** Access control enforced inside Elasticsearch means application code cannot accidentally leak cross-tenant data. The filter applies regardless of which query path retrieves the document.

**Ecosystem maturity.** Kibana, Logstash, Beats, and 15 years of operational tooling surround it. Monitoring, alerting, snapshot management, and index lifecycle policies exist and are documented.

## Critical Limitations

**Concrete failure mode — relevance degradation under high dimensionality.** HNSW approximate nearest neighbor search trades recall for speed via the `num_candidates` and `ef_construction` parameters. At default settings with large embedding dimensions (1536+ from OpenAI models), recall at top-10 can drop below 90% for queries in sparsely populated regions of the embedding space. This manifests as retrievals that miss clearly relevant documents without surfacing any error. Tuning `num_candidates` upward restores recall but increases query latency proportionally.

**Unspoken infrastructure assumption — JVM memory management.** Elasticsearch requires `vm.max_map_count >= 262144` on Linux (explicit in RAGFlow's setup instructions) and allocates 50% of system RAM to the JVM heap by default, with the other 50% reserved for Lucene's off-heap page cache. On a 16GB host this means 8GB to the JVM, leaving limited room for co-located services. Deployments that undersize RAM relative to index size hit garbage collection pauses that appear as intermittent query latency spikes rather than explicit errors.

## When NOT to Use It

Skip Elasticsearch when:

- **You need local-first, no-infrastructure retrieval.** SQLite-backed vector search (sqlite-vec, DuckDB) or in-process FAISS handles single-user or developer workloads at zero operational cost. Elasticsearch requires Docker, JVM, and memory you may not have.
- **Your corpus fits in memory and latency is the primary constraint.** In-process FAISS or Annoy with a simple inverted index will beat Elasticsearch on p50 latency by 5-10x for corpora under ~1M documents, with no network round-trip.
- **You cannot afford index management overhead.** Shard configuration, replica counts, index lifecycle policies, and mapping freezes require ongoing attention. A team without dedicated infrastructure capacity will spend time on cluster health rather than product.
- **Your query patterns are purely semantic with no keyword or metadata filtering.** A purpose-built vector database (Weaviate, Qdrant, Pinecone) may offer better vector-specific performance and simpler operations when BM25 and document-level security aren't required.

## Unresolved Questions

**Cost at scale.** Elastic's licensing changed in January 2021 (SSPL/Elastic License 2.0), forking from AWS OpenSearch. Self-hosted remains free; Elastic Cloud pricing scales with compute and storage in ways that become significant at tens of millions of documents. No independent cost benchmarks for large RAG deployments appear in public literature.

**ELSER model update cadence.** ELSER is Elastic's proprietary sparse encoder. How frequently it retrains, what happens to existing indices when a new version releases (re-indexing required or not), and whether the model weights can be extracted for offline use are not clearly documented in public resources.

**Hybrid retrieval score normalization.** When combining BM25 and vector scores, the relative weighting significantly affects which documents surface. Elasticsearch's reciprocal rank fusion mode partially addresses this, but optimal weighting for specific domains requires empirical tuning. The documentation offers guidance; the optimal answer for any given corpus remains workload-specific.

**Multi-tenancy isolation guarantees.** Document-level security filters apply at query time via role-based query fragments. Whether a misconfigured role definition can leak cross-tenant documents, and what audit mechanisms exist to detect such leaks, are underspecified in operational documentation.

## Alternatives

| Need | Use instead |
|---|---|
| Local development / single user | sqlite-vec, FAISS, ChromaDB |
| AWS-native deployment | OpenSearch (Elasticsearch fork, API-compatible) |
| Pure vector workloads, no BM25 needed | Qdrant, Weaviate, Pinecone |
| Relational + vector in one system | pgvector on PostgreSQL |
| RAGFlow default, lower infrastructure overhead | Infinity (RAGFlow's purpose-built alternative) |
| Full Elasticsearch feature set, open license | OpenSearch (Apache 2.0) |

Use Elasticsearch when you need hybrid BM25+vector retrieval, document-level security, and can support the JVM-based infrastructure. OpenSearch is a drop-in alternative if the Elastic license is a concern. For pure vector search at smaller scale, the dedicated vector databases offer simpler operations and comparable recall.
