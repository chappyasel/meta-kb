# ChromaDB

## What It Does

ChromaDB is an open-source vector database purpose-built for LLM applications. It stores document embeddings alongside metadata and text, then retrieves the most semantically similar items for a query. The core value proposition is simplicity: a developer can go from `pip install chromadb` to a working semantic search system in under ten lines of Python.

Chroma runs in three modes: in-memory (ephemeral, for testing), persistent local (SQLite + HNSW on disk), and client-server (for production deployments). The client-server mode uses a FastAPI HTTP backend.

Within the agent infrastructure space, Chroma appears most often as the vector storage layer in RAG pipelines, the backing store for agent memory systems, and as the semantic deduplication engine in training pipelines. The AgentEvolver codebase, for example, uses ChromaDB inside its `StateRecorder` class to detect when an explorer agent revisits semantically similar environment states during task synthesis. Claude-mem lists ChromaDB as one of its vector search backends (alongside SQLite FTS5) for the optional semantic search layer.

## Architecture

Chroma's storage layer uses two components together:

- **SQLite** for metadata, document text, and collection management
- **HNSW (Hierarchical Navigable Small World)** via the `hnswlib` library for the approximate nearest-neighbor index

Documents are stored in SQLite, and their embeddings live in an HNSW index that Chroma maintains on disk as a set of binary files. Metadata filtering happens at the SQLite layer before or after vector search, depending on the query.

Collections are the primary organizational unit. Each collection holds documents, their embeddings (either provided by the caller or generated by a configured embedding function), and arbitrary metadata key-value pairs.

The embedding function interface is pluggable. Chroma ships built-in integrations for OpenAI, Cohere, HuggingFace, and sentence-transformers, with a fallback to `all-MiniLM-L6-v2` via sentence-transformers when no function is specified. For agents that embed their own text externally, callers can pass raw embedding arrays directly.

Query operations support both vector similarity search and metadata filtering via a `where` clause (MongoDB-style JSON operators). Hybrid queries combining both are possible but not deeply optimized — the metadata filter runs as a post-processing step over vector candidates.

## Strengths

**Zero-infrastructure start.** Running `chromadb.Client()` creates an in-memory database with no configuration. Switching to persistent mode is one argument change. This frictionless onboarding explains its prevalence in tutorials and agent prototypes.

**Metadata filtering.** The `where` clause lets callers filter by arbitrary metadata before or alongside vector search. For agent memory systems where memories are scoped by `user_id`, `session_id`, or `agent_id` (as in Mem0's architecture), this metadata filtering is essential and works without a separate database.

**Embedding function abstraction.** The pluggable embedding interface means calling code does not need to manage the embed-then-store pipeline manually. Collections can be created with a specific embedding function and Chroma handles embedding on insert and query.

**Python-native API.** The API maps cleanly to how Python developers think: collections as objects, `add()` / `query()` / `delete()` as methods, results as typed dictionaries.

## Critical Limitations

**Concrete failure mode — HNSW at scale.** HNSW indices do not support efficient deletion. When a document is deleted from a Chroma collection, the vector is soft-deleted from SQLite but the HNSW index entry remains until the index is rebuilt. At high delete rates (common in agent memory systems that update or invalidate memories frequently), the index accumulates ghost entries that degrade both recall and memory usage. Mem0's memory reconciliation pipeline, which issues frequent DELETE operations as facts are superseded, would encounter this problem at production scale.

**Unspoken infrastructure assumption — local disk is durable.** The default persistent mode writes HNSW binary files to a local directory. There is no replication, no backup, and no write-ahead log beyond SQLite's journal. Containerized deployments that do not mount a persistent volume will lose all vector data on restart. This assumption is fine for local development but breaks silently in Kubernetes or ephemeral cloud environments without explicit volume configuration.

## When NOT to Use It

**High-volume production RAG.** When query latency SLAs matter, index size exceeds a few million vectors, or the application needs filtered search performance guarantees, Chroma's single-node HNSW is the wrong choice. Purpose-built managed services — Pinecone, Qdrant, Weaviate — offer distributed indices, horizontal scaling, and operational tooling that Chroma does not.

**Frequent update/delete workloads.** Agent memory systems that continuously add, update, and delete memories (like Mem0's ADD/UPDATE/DELETE reconciliation loop) will degrade Chroma's index over time due to the HNSW ghost entry problem described above. Qdrant and Weaviate handle deletions more cleanly.

**Multi-tenant production deployments.** Chroma's collection model does not offer tenant isolation, access control, or quota management. All collections in a Chroma instance share the same HNSW file and SQLite database. For multi-user agent applications, this requires application-level workarounds (separate collections per user, or separate Chroma instances).

**When you need hybrid search.** BM25 + vector hybrid search, as described in the [Hybrid Search](../concepts/hybrid-search.md) concept, is not a first-class Chroma feature. Pipelines requiring lexical+semantic retrieval combination need either a different database or an external BM25 layer.

## Unresolved Questions

**Governance and maintenance trajectory.** Chroma Inc. raised a Series A in 2023, but the long-term sustainability and roadmap (especially relative to the rapidly evolving managed vector DB market) is not clearly documented. The company's commercial offering (`chromadb.com`) runs alongside the open-source project, but the boundary between open-source and commercial features is opaque.

**Cost at scale.** The documentation does not address what happens operationally when a Chroma deployment grows to tens of millions of vectors. Index rebuild times, RAM requirements for HNSW, and disk I/O characteristics at scale are not published. Teams find out empirically.

**Conflict resolution in concurrent writes.** The client-server mode uses FastAPI with async handlers. The documentation does not clearly specify what happens to HNSW consistency under concurrent writes from multiple clients. SQLite's WAL mode handles the metadata layer, but HNSW's in-memory index update behavior under concurrency is not documented.

**Embedding function versioning.** When the embedding model for a collection changes (a common event as better models are released), there is no built-in migration path. All vectors must be re-embedded from scratch. This is a practical problem for long-lived agent memory stores.

## Alternatives

**[FAISS](../projects/faiss.md)** — Use when you need maximum raw ANN performance, GPU acceleration, or research-grade index variants (IVF, PQ, etc.). FAISS is a library, not a database — no metadata storage, no persistence layer. Best for offline batch retrieval or when you manage your own storage.

**[Qdrant](../projects/qdrant.md)** — Use when you need production-grade vector search with clean deletion semantics, payload filtering, sparse+dense hybrid search, and horizontal scaling. Qdrant's Rust implementation is significantly faster than Chroma at comparable scales and handles high update rates correctly.

**Pinecone** — Use when you want fully managed infrastructure with zero operational burden, are comfortable with a proprietary SaaS pricing model, and need production-grade uptime guarantees. Higher cost but eliminates infrastructure management.

**[PostgreSQL](../projects/postgresql.md) with pgvector** — Use when the application already runs on Postgres, vector search volume is moderate (under a few million vectors), and operational simplicity (one database for everything) is a priority. pgvector supports HNSW and IVF indices directly in Postgres with transactional consistency.

**[LlamaIndex](../projects/llamaindex.md) or [LangChain](../projects/langchain.md) abstractions** — Both frameworks abstract the vector store interface, letting applications swap Chroma for Qdrant or Pinecone without rewriting retrieval logic. Starting with Chroma via these abstractions and migrating later is a viable path.

## Relationship to Agent Infrastructure

ChromaDB sits at the [Vector Database](../concepts/vector-database.md) layer of agent infrastructure, serving [Semantic Memory](../concepts/semantic-memory.md) and [Retrieval-Augmented Generation](../concepts/retrieval-augmented-generation.md) use cases. Most [Agent Memory](../concepts/agent-memory.md) systems that use vector storage — [Mem0](../projects/mem0.md), claude-mem's optional semantic layer, AgentEvolver's state deduplication — could use Chroma as a drop-in backend, though production deployments of these systems often migrate to more capable backends.

The [Semantic Search](../concepts/semantic-search.md) pattern that Chroma enables is foundational to [Agentic RAG](../concepts/agentic-rag.md) architectures, where agents retrieve context on-demand rather than loading full conversation histories into context windows.
