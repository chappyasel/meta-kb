# SkillWeaver

## What It Does

SkillWeaver generates, evaluates, and assembles agent skills automatically. An agent executes tasks, and SkillWeaver analyzes the resulting traces to extract reusable skill definitions, write them to a persistent skill library, and load them into future task contexts. This sits squarely in the [Self-Improving Agent](../concepts/self-improving-agent.md) paradigm: model weights stay frozen, and all capability accumulation happens in external memory.

The distinguishing architectural claim is "weaving" -- skills are not just accumulated individually but composed together. When a new task arrives, SkillWeaver retrieves relevant skills from the library and combines them, creating emergent capabilities from existing primitives rather than always generating from scratch.

SkillWeaver is documented as a reference implementation within Anthropic's skills ecosystem and appears as a baseline provider in the MemEvolve benchmark suite, which reimplemented it alongside eleven other memory systems for comparison purposes. [Source](../raw/deep/repos/bingreeky-memevolve.md)

---

## Architecture

SkillWeaver's core loop:

1. **Execute**: Run tasks through the agent, logging full execution traces.
2. **Extract**: Analyze traces to identify reusable action sequences worth abstracting into skills.
3. **Validate**: Test candidate skills against held-out tasks before committing them to the library.
4. **Store**: Write validated skills to a persistent library (typically SKILL.md format with YAML frontmatter).
5. **Retrieve**: On new tasks, search the library using hybrid BM25 + semantic retrieval.
6. **Weave**: Compose multiple retrieved skills into a combined context before task execution.

The SKILL.md format follows the [Agent Skills](../concepts/agent-skills.md) specification hosted at agentskills.io -- `name` and `description` fields in YAML frontmatter, with instructions in the markdown body. This means skills generated by SkillWeaver can be loaded by any compliant consumer, including [Claude](../projects/claude.md) via the plugin marketplace. [Source](../raw/deep/repos/anthropics-skills.md)

Retrieval uses the same three-tier progressive disclosure architecture described in Anthropic's reference implementation: description metadata always in context, full SKILL.md loaded on trigger, bundled resources loaded on demand. [Source](../raw/deep/repos/anthropics-skills.md)

## How Skill Composition Works

Standard skill systems load one skill per task. SkillWeaver's differentiator is loading *multiple* relevant skills and combining them. The mechanism:

- Retrieve top-k skills from the library ranked by relevance.
- Concatenate their instruction bodies into a combined skill context.
- Present this composed context to the agent before task execution.

This enables solving tasks that require capabilities from multiple prior experiences -- a skill for API authentication composed with a skill for data transformation, for instance. The composition is additive and happens in the context window, not through any programmatic skill-to-skill interface.

This is simpler than [Skill Composition](../concepts/skill-composition.md) via explicit dependency graphs, but it works within the constraints of the [Agent Skills](../concepts/agent-skills.md) spec, which has no inter-skill communication mechanism. See also [Agent Workflow Memory](../projects/agent-workflow-memory.md) for a related approach that uses workflow graphs instead of composed instruction contexts.

## Relationship to the Anthropic Ecosystem

[Anthropic](../projects/anthropic.md) built SkillWeaver as part of the broader agent skills infrastructure around [Claude](../projects/claude.md). The governance layer (naming conventions, marketplace distribution, description-driven triggering, the skill-creator meta-skill for eval-driven iteration) comes from the anthropics/skills repository, which SkillWeaver sits inside. [Source](../raw/deep/repos/anthropics-skills.md)

The self-evaluation loop -- where a skill-creator subagent runs test prompts, grades outputs with assertions and LLM-as-judge, and iterates on skill quality -- applies to skills SkillWeaver generates. Without this loop, skills extracted from traces might optimize for past tasks but generalize poorly.

Harrison Chase's framing is useful here: SkillWeaver operates at the *context layer* of continual learning (not model weights, not harness code), which gives it faster feedback loops than retraining but also means improvements are bounded by what the base model can articulate and generalize. [Source](../raw/tweets/hwchase17-continual-learning-for-ai-agents.md)

## Strengths

**Deployment-time learning with no retraining cost.** Skills accumulate from live task executions. Operators get improving capability without touching model parameters or paying fine-tuning costs.

**Format portability.** Skills in SKILL.md format work across [Claude](../projects/claude.md) Code, Claude.ai, and the API. Generated skills can be inspected, edited, and redistributed through the agentskills.io marketplace.

**Composability from an existing library.** Weaving multiple skills together handles tasks that require combining prior experiences -- a meaningful improvement over single-skill retrieval.

**Eval-driven quality control.** The skill-creator meta-skill provides a structured loop (generate, test with realistic prompts, grade with assertions, iterate, optimize descriptions for reliable triggering) that catches poor skills before they enter the library. [Source](../raw/deep/repos/anthropics-skills.md)

## Critical Limitations

**Concrete failure mode -- description triggering brittleness.** Skill retrieval depends on the agent semantically matching a task description to a skill description. Anthropic's own skill-creator documentation explicitly warns about undertriggering: Claude tends not to invoke skills when they would be useful. Skills extracted from traces inherit whatever description the extraction process wrote. If that description is too narrow or uses different vocabulary than future tasks, the skill never fires. The full eval loop (20 queries × 3 runs × 5 iterations = 300 LLM calls) exists specifically because this is hard to get right, and it runs at development time, not continuously in production. [Source](../raw/deep/repos/anthropics-skills.md)

**Unspoken infrastructure assumption -- trace quality determines skill quality.** SkillWeaver extracts skills from execution traces. If agent traces contain suboptimal paths (retries, dead ends, workarounds), extracted skills capture those patterns. There is no mechanism to distinguish a clean trace from a lucky one. Systems like MemEvolve address this with explicit trajectory analysis and temperature-controlled generation; SkillWeaver's extraction fidelity is bounded by trace quality directly. [Source](../raw/deep/repos/bingreeky-memevolve.md)

## When Not to Use It

**Tasks requiring architectural adaptation, not instruction accumulation.** SkillWeaver adds skills as instruction contexts. If the agent's memory architecture itself is the bottleneck -- not the instructions it receives -- SkillWeaver cannot help. [MemEvolve](../projects/memevolve.md) addresses this by generating entirely new memory provider code; SkillWeaver cannot. [Source](../raw/deep/repos/bingreeky-memevolve.md)

**High-reliability domains with no tolerance for skill regression.** There is no mechanism for ongoing trigger-rate monitoring in production. A skill added to the library can degrade other skills' triggering through context competition, and there is no automated alert for this.

**Multi-agent pipelines where skills need to communicate.** The agentskills.io spec has no inter-skill communication. Skills cannot pass structured outputs to each other programmatically. If your architecture requires one skill's result to change another skill's behavior, you need a different approach -- [LangGraph](../projects/langgraph.md) for explicit data flow graphs, or [Memento-Skills](../projects/mem0.md) for the playbook execution mode. [Source](../raw/deep/repos/memento-teams-memento-skills.md)

**Rapidly evolving task domains.** Skills written against past traces can become stale. The framework has no built-in mechanism to deprecate outdated skills or detect when a previously valid skill is no longer appropriate. This is [Continual Learning](../concepts/continual-learning.md)'s version of knowledge base decay.

## Unresolved Questions

**How does SkillWeaver handle skill conflict?** When two retrieved skills give contradictory instructions about the same operation, what happens? The agentskills.io spec and the anthropics/skills repo have no conflict resolution mechanism. [Source](../raw/deep/repos/anthropics-skills.md)

**What is the cost at scale?** A skill library that grows from hundreds of tasks could reach hundreds of skills. At some point the retrieval problem becomes non-trivial, context budgets get stressed when multiple skills are loaded, and the description optimization loop (300 LLM calls per skill) becomes expensive. There is no published guidance on library size limits or retrieval degradation.

**How does SkillWeaver's extraction distinguish task-specific workarounds from generalizable patterns?** The source material describes skill extraction from traces but not the criteria for deciding what constitutes a reusable skill versus a one-off solution. This matters: libraries polluted with overfitted skills degrade retrieval precision.

**Does the composition step have a token budget?** Weaving multiple skills concatenates their instruction bodies. For skills with large reference files, this could exhaust available context before the task even starts. The three-tier progressive disclosure architecture nominally addresses this, but the composition logic's interaction with per-skill context budgets is not documented. [Source](../raw/deep/repos/anthropics-skills.md)

## Alternatives

**Use [Voyager](../projects/voyager.md) when** you need self-expanding skills in a closed-world environment with verifiable task completion criteria. Voyager's skill library was validated in Minecraft where ground-truth success is unambiguous.

**Use [Agent Workflow Memory](../projects/agent-workflow-memory.md) when** tasks have structured, repeatable workflows. AWM extracts workflow graphs rather than instruction text, which provides more structured composition and avoids the undertriggering problem.

**Use [MemEvolve](../projects/memevolve.md) when** the memory architecture itself needs to change across task domains, not just the instruction content within a fixed architecture. MemEvolve generates new memory provider code from traces; SkillWeaver generates new instruction files.

**Use [Memento-Skills](../projects/mem0.md) when** you need deployment-time skill evolution with explicit failure-driven rewriting, formal utility scoring, and execution in sandboxed playbook mode. Memento-Skills implements the same read-execute-reflect-write loop but with more rigorous skill governance. [Source](../raw/deep/repos/memento-teams-memento-skills.md)

**Use [DSPy](../projects/dspy.md) when** you want automated optimization of prompts and instructions with formal metrics, rather than trace-based skill extraction.
