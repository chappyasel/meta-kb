# Generative Agents

**Type:** Research Project | **Domain:** Agent Memory & Simulation | **Origin:** Stanford University (Park et al., 2023)

---

## What It Is

Generative Agents is a Stanford research paper and accompanying simulation where 25 LLM-powered virtual characters live, work, and interact in a Sims-like sandbox environment called Smallville. Each agent maintains an independent sense of identity, relationships, and daily schedule. The core claim: you can produce convincingly human-like social behavior by combining LLMs with three specific memory mechanisms rather than relying on ever-larger context windows or hand-scripted behavior trees.

The paper was published at USENIX SOUPS 2023 (ACM UIST). The research code is available at [joonspk-research/generative_agents](https://github.com/joonspk-research/generative_agents).

---

## Architecture: How It Works

The architecture has four interlocking components, each with a concrete implementation:

**1. Memory Stream**

The foundational data structure. Every agent maintains a flat list of *memory objects*, each storing:
- Natural language description of an observation or event
- Creation timestamp and most-recent-access timestamp
- Importance score (an integer 1–10, generated by querying the LLM on first storage)

Memory objects get written constantly: every time an agent perceives something in the environment, a new record is appended. This is append-only — nothing gets deleted or overwritten.

**2. Retrieval Function**

When an agent needs to act or respond, a scoring function surfaces the most relevant memories. The retrieval score for each memory object is:

```
score = α_recency × recency + α_importance × importance + α_relevance × relevance
```

- **Recency**: exponential decay from last-access time (decay factor ~0.995 per hour of game time)
- **Importance**: the stored 1–10 LLM-generated score
- **Relevance**: cosine similarity between the memory's embedding and the current query embedding

All three components are normalized to [0,1] before summing. The top-k scored memories get injected into the LLM prompt as context. This replaces the need for a full context window — only the most salient memories surface at any given moment.

**3. Reflection**

Periodically (triggered when the sum of recent memory importance scores crosses a threshold), agents run a reflection pass. The process:
1. Take the 100 most recent memory objects
2. Ask the LLM: "Given these statements, what are the 3 most salient high-level questions you can answer?"
3. For each question, retrieve relevant memories and ask the LLM to synthesize an insight
4. Store those insights back into the memory stream as new memory objects, marked as reflections

Reflections are higher-order abstractions. An agent might store "Klaus is worried about his job" as a reflection derived from a dozen individual observations about Klaus's behavior. These reflections then influence future retrieval alongside raw observations.

**4. Planning**

Each morning (in simulated time), agents generate a daily plan in broad strokes using a prompt template: `{Agent description}. Here is {Agent name}'s plan today in broad strokes: 1)`. These plans decompose recursively into hourly, then 5-minute chunks. Plans get revised reactively when new observations warrant it — if an agent encounters something unexpected, they re-evaluate whether to continue their plan or respond.

**Agent Interaction**

When two agents meet, the system constructs a dialogue prompt containing both agents' retrieved memories about each other and the current context. Dialogue continues until both agents signal completion. New statements from dialogue get written back into each agent's memory stream.

## Key Numbers

- **25 agents** simulated simultaneously in Smallville
- **GPT-3.5-turbo** used as the underlying LLM (at time of publication)
- **2-day simulated time** in the main experiment (real-world wall time: several hours of inference)
- **Human evaluation**: raters assessed agent responses as more believable than ablated versions (no reflection, no planning, no memory stream); self-reported by the authors, not independently replicated at scale

The paper includes an ablation study comparing full architecture against versions missing each component. Agents without reflection produced notably less coherent long-term behavior. These results are self-reported — the evaluation relies on human judges recruited by the authors rating plausibility on Likert scales. No third-party replication of the full simulation has been published as of this writing.

## Strengths

**Emergent social dynamics without scripting.** Information propagates between agents organically — one agent tells another about an event at the coffee shop, and that information spreads through the community in ways the researchers did not program. A Valentine's Day party planned by one agent attracted others who autonomously decided to attend and invite their own contacts. This emergence comes from the architecture, not from hard-coded social rules.

**Scalable memory without full context.** The retrieval scoring function solves a real problem: how to give an agent access to months of memories without blowing a context window. By ranking on recency × importance × relevance, the system ensures only situationally relevant memories surface. This is more principled than naive RAG (which ranks only on relevance) because it accounts for both staleness and intrinsic salience.

**Reflection produces coherent identity over time.** Without reflection, agents tend to respond to immediate context without a persistent sense of self. With reflection, synthesized higher-order beliefs ("I value my relationship with John") persist in the memory stream and influence behavior days later in simulated time.

## Critical Limitations

**Concrete failure mode — retrieval scoring brittleness:** The importance score is assigned once at memory creation by an LLM call. If the model misjudges importance at write time (rating a mundane event as 8/10 or a critical one as 3/10), that error compounds over the agent's lifetime. The score never updates. A memory about a trivial conversation written during a context where the LLM was confused could permanently outrank a genuinely important memory in retrieval. There is no correction mechanism.

**Unspoken infrastructure assumption:** The simulation requires a live, callable LLM endpoint for every perception-action cycle of every agent. With 25 agents each potentially triggering several LLM calls per simulated minute (observations, planning checks, reflections, dialogue), the API cost and latency at real-time simulation speeds would be prohibitive. The published experiments ran slower than real time, which obscures what "interactive" actually means operationally. Anyone trying to deploy this for real-time game NPCs or interactive applications at scale would hit hard throughput and cost walls. The paper does not address this.

## When Not to Use It

**Don't use Generative Agents when:**

- You need real-time NPC behavior in games — the LLM call overhead per agent per tick makes this infeasible at playable frame rates with current inference speeds
- Your agents need precise, factual memory recall — the retrieval function can bury accurate old memories under higher-scored but less relevant ones, and reflections can introduce synthesis errors that persist as "memories"
- You're operating under strict cost budgets — 25 agents over a 2-day simulation generates hundreds of thousands of tokens in LLM calls
- You need reproducible, deterministic behavior — LLM stochasticity means the same setup will produce meaningfully different social outcomes across runs
- Your application requires agents to forget or correct specific memories — the append-only memory stream has no targeted deletion mechanism

## Unresolved Questions

The paper does not explain:

- **How reflection frequency was calibrated.** The importance-sum threshold that triggers reflection is set empirically but not justified — what happens if it fires too often or too rarely at scale?
- **Conflict resolution between contradictory memories.** If agent A tells agent B something false, and B later learns the truth, both observations sit in B's memory stream. The retrieval function might surface either one depending on context. There is no explicit mechanism for resolving contradictions.
- **Scaling to larger agent populations.** All experiments use exactly 25 agents. The paper does not analyze how retrieval quality, planning coherence, or social emergence change with 100 or 1000 agents.
- **Cost and latency at real-time interactive speeds.** The published supplementary materials note inference ran over extended wall-clock time. No cost figures appear.
- **What happens when underlying model capability changes.** The architecture's plausibility depends heavily on GPT-3.5-turbo's ability to assign sensible importance scores and synthesize coherent reflections. Behavior with weaker or stronger models is untested in the paper.

## Relationship to Adjacent Work

Generative Agents sits in a cluster of memory-augmented agent architectures. The mem-agent paper from Dria (2025) cites it as an example of "semi-episodic memory in a declarative format" — observing that the memory stream stores LLM-generated summaries rather than raw trajectories, making it more declarative than truly episodic. [Source](../../raw/articles/hugging-face-mem-agent-equipping-llm-agents-with-memory-using.md)

Lilian Weng's 2023 survey of LLM agents describes the retrieval model's three-factor scoring as a clean instantiation of the general pattern: importance distinguishes mundane from core memories, recency weights recent events higher, relevance grounds retrieval in the current situation. [Source](../../raw/articles/lil-log-llm-powered-autonomous-agents.md)

**Alternatives and selection guidance:**

- **MemGPT / Letta**: Use when you need an explicit working-memory / long-term-memory distinction with agent-controlled promotion between tiers, rather than a flat scored stream
- **Reflexion**: Use when the goal is iterative task improvement through verbal self-critique on individual trajectories, rather than persistent social simulation
- **mem-agent**: Use when you need a deployable, fine-tuned small model (4B parameters) for personal-assistant memory management rather than a multi-agent social simulation scaffold
- **Voyager**: Use when the domain is open-ended skill acquisition (procedural memory) rather than episodic/declarative social memory
