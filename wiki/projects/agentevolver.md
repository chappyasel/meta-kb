# AgentEvolver

**Repository:** [modelscope/AgentEvolver](https://github.com/modelscope/AgentEvolver) | 1,336 stars, 152 forks | Apache-2.0

AgentEvolver is a self-evolving agent training framework from ModelScope (Alibaba). It closes the loop between task generation, policy rollout, and optimization without requiring human-annotated training data. Three mechanisms work together: agents explore environments to synthesize their own tasks (Self-Questioning), retrieve distilled past experience during rollouts (Self-Navigating), and receive step-level reward signals calibrated by an LLM judge (Self-Attributing). The technical differentiator over standard [GRPO](../concepts/grpo.md) training is the ADCA-GRPO credit assignment scheme, which replaces trajectory-level outcome rewards with dense process signals that reportedly cut training steps by ~40%.

## What It Does

Standard agent RL training requires three expensive ingredients: a labeled task dataset, a functioning rollout environment, and a reward signal. AgentEvolver automates all three. The task dataset is generated by having the agent explore the environment under a high-temperature policy, then summarizing trajectories into structured task specs. Experience from successful and failed prior runs gets indexed in a vector store and retrieved during future rollouts. Credit assignment uses a separate LLM to label each step GOOD or BAD, fusing those labels with terminal outcome rewards before computing advantages.

The result is a training loop that can, in principle, run indefinitely without human data curation -- the agent generates harder tasks as it improves, accumulates experience, and receives more informative gradients. [Self-Improving Agents](../concepts/self-improving-agents.md) is the broader concept; AgentEvolver is one of the more complete open implementations of it.

## Core Architecture

The codebase under `agentevolver/module/` has six primary subsystems:

**Task Manager** (`task_manager/`) runs a `LlmRandomSamplingExploreStrategy` that dispatches an exploration agent with a 118-line structured prompt covering six phases of progressive environment probing. Post-exploration, `_step_summarize()` converts action-observation trajectories into `<task>` blocks with a query, confidence score, and reference solution. Two filter stages remove duplicates: `NaiveTaskPostFilter` (lexical) and an `EmbeddingClient` backed by ChromaDB (semantic, via cosine similarity). The `FullDataset` class manages the complete data lifecycle and mixes synthetic tasks with original environment tasks via a `MixtureStrategy`.

**Experience Manager** (`exp_manager/exp_manager.py`) pairs with an external ReMe service (HTTP). After rollouts, `summarize_in_batch()` ships trajectories to ReMe for extraction and vector indexing. During rollouts, `ExperienceWorker.manage_rollout_context()` retrieves top-k experiences and prepends them with a configurable template. The `allocate_train_mode()` method controls whether experience tokens survive into training -- the default `"alldiscard"` strips them to prevent the policy from becoming dependent on external memory at inference time.

**Advantage Processor** (`adv_processor/`) contains the ADCA-GRPO implementation. `evaluate_step_flags_parallel()` in `semantic_attribution.py` sends one API call per trajectory (not per step) to an LLM judge (default: `qwen-max` via DashScope), extracting "Step N Judgment: GOOD/BAD" patterns. `compute_prm_grpo_advantages()` in `adca_grpo.py` then:
1. Maps GOOD → `+fix_base` (0.2), BAD → `-fix_base`
2. Z-score normalizes step-level PRM rewards and trajectory-level outcome scores independently
3. Fuses them: `r_t = alpha * r_t_attr + indicator(t=T) * r_out`
4. Computes suffix sums per step via `suffix_sum_on_steps()`
5. Broadcasts step advantages to tokens via `broadcast_step_adv_to_tokens()` using a `step_ids` tensor

Alpha supports cosine decay (`alpha_t = alpha_min + 0.5 * (alpha0 - alpha_min) * (1 + cos(pi * p))`), implementing curriculum learning where process signals dominate early and fade as the agent matures.

**Step Parser** (`step_parser.py`) is critical infrastructure. `parse_response_ids_to_steps()` extracts role-header tokens for any model family by comparing tokenizations of messages with and without content, then builds a `StepParseResult` mapping each response token to its containing step. This model-agnostic design handles Qwen, Llama, and other chat templates. `verify_step_alignment()` catches mismatches between semantic evaluation step counts and advantage scaling step counts -- a class of bugs that would silently corrupt training.

**Environment Service** (`env_service/`) runs as a separate FastAPI server backed by Ray actors, exposing `/create`, `/step`, `/evaluate`, and `/release`. Environments (AppWorld, BFCL, WebShop, OpenWorld) register via `import_and_register_env()` in `env_service/registry.py`. Idle instances clean up after 3600 seconds.

**Training Loop** (`ae_ray_trainer.py`) extends veRL's `RayPPOTrainer`. `ParallelEnvManager.rollout()` manages thread-level progress via a `tmux` dict tracking step and token counts per thread. The `samples_to_dataproto()` method converts trajectories into `DataProto` tensors with custom fields: `step_ids`, `group_ids`, `exp_mask`. Failed tasks retry with exponential backoff (up to 4 attempts).

## Benchmarks

Tested on AppWorld (multi-step API interactions) and BFCL v3 (multi-turn function calling). Results are self-reported by the authors; no independent verification is available.

| Model | Config | AppWorld avg@8 | BFCL v3 avg@8 | Combined avg@8 |
|---|---|---|---|---|
| Qwen2.5-7B | Baseline | 1.8 | 29.8 | 15.8 |
| Qwen2.5-7B | +Questioning | 23.2 | 49.0 | 36.1 |
| Qwen2.5-7B | +Q+Navigating | 26.3 | 53.3 | 39.8 |
| Qwen2.5-7B | +Q+Attributing | 25.7 | 56.8 | 41.3 |
| Qwen2.5-7B | Full | 32.4 | 57.9 | 45.2 |
| Qwen2.5-14B | Baseline | 18.0 | 41.6 | 29.8 |
| Qwen2.5-14B | Full | 48.7 | 66.5 | 57.6 |

Three findings from the ablation worth noting: Self-Questioning alone accounts for the majority of the gain (15.8 → 36.1 for 7B), suggesting the primary bottleneck was training data quality. Self-Attributing adds more than Self-Navigating at both model sizes. The 14B model benefits more from each incremental mechanism, suggesting the techniques require sufficient model capacity.

Training used 8 × A100 80GB, lr=1e-6, batch size 32, 40 epochs per update, KL penalty 0.001, max 30 steps per trajectory.

## Strengths

**Closed-loop training without human annotation.** The Self-Questioning mechanism generates, curates, and deduplicates tasks entirely automatically. The `FullDataset` class checkpoints task generation with MD5-based cache invalidation, making the pipeline resumable.

**Dense reward signals for long-horizon tasks.** Sparse terminal rewards fail when trajectories span 10-30 steps with only a binary success signal at the end. ADCA-GRPO provides a reward at every step, and the independent z-score normalization prevents terminal rewards from drowning process signals.

**Model-agnostic step parsing.** `_extract_role_header_tokens()` in `step_parser.py` works by inspecting tokenization differences rather than hardcoding chat template strings. This is directly reusable in any multi-turn RL training codebase.

**Pluggable environment abstraction.** Any environment implementing `get_init_state()`, `step()`, `evaluate()`, and `close()` integrates via `Registry.register()` without touching the training code.

## Critical Limitations

**DashScope API lock-in.** The semantic attribution system and task manager are hardcoded to `https://dashscope.aliyuncs.com/compatible-mode/v1` with `DASHSCOPE_API_KEY`. The code raises `RuntimeError` if the key is absent. The API format is OpenAI-compatible, so adaptation is technically straightforward, but it requires code modification -- not a configuration change. Any non-Alibaba deployment needs to patch `semantic_attribution.py` and `task_manager.py`.

**Unverified 40% training step reduction.** The claim that ADCA-GRPO reduces training steps by ~40% versus standard GRPO appears in the technical summary but is not directly reproducible from the released code without running both configurations on identical hardware with identical seeds. The released benchmarks show endpoint performance, not convergence curves.

**Unspoken infrastructure assumption: GPU cluster with Ray.** The framework requires veRL (Bytedance's distributed RL library), Ray, and 8 × A100 80GB for the reported experiments. Single-GPU or CPU-only deployments are not supported. The service-oriented architecture (five separate services: environment, ReMe, logview, training, crafters) assumes stable inter-process networking and adds operational complexity proportional to the number of target environments.

## When Not to Use This

**Inference-only or prompt-engineering workflows.** AgentEvolver modifies model weights through RL fine-tuning. If you need to improve agent behavior without training (through [Prompt Optimization](../concepts/prompt-optimization.md), better [Context Engineering](../concepts/context-engineering.md), or [Retrieval-Augmented Generation](../concepts/retrieval-augmented-generation.md)), this is the wrong tool.

**Environments without automated evaluation.** The Self-Questioning mechanism requires the environment to programmatically evaluate whether a task was completed. Human-in-the-loop evaluation (see [Human-in-the-Loop](../concepts/human-in-the-loop.md)) breaks the autonomous training loop. AppWorld and BFCL v3 have clean success signals; document Q&A or creative writing tasks do not.

**Teams without GPU infrastructure.** The minimum viable setup is 8 × A100 80GB. Smaller setups will require significant adaptation of the veRL/Ray configuration.

**Production deployment of the trained agent.** The framework is a training system, not a deployment framework. The trained model weights are the output; serving infrastructure is out of scope.

## Unresolved Questions

The documentation doesn't address several operational concerns:

- **LLM judge cost at scale.** Semantic attribution calls `qwen-max` once per trajectory. With 32 trajectories per batch and 40 epochs per update, the API cost per training run is non-trivial. The `prm_steps` parameter disables attribution after N epochs as a cost control, but the right value for this parameter is not explained.

- **Experience pool quality over time.** The default `updated_freq=0` populates the experience pool only at initialization. Whether continuously updated experience pools improve or degrade performance is not analyzed. Experience retrieval quality is unverified at runtime -- irrelevant retrieved experiences could harm rollout quality with no detection mechanism.

- **Task generation stability.** The exploration prompt encourages diverse environment coverage, but there is no mechanism to verify that generated tasks are uniformly distributed across the difficulty spectrum. The ablation shows task quality matters more than the optimization algorithm; the conditions under which task generation degrades are not characterized.

- **Single exploration strategy.** `get_exploration_strategy()` raises `NotImplementedError` for anything except `LlmRandomSamplingExploreStrategy`. The roadmap mentions "cross-stage collaborative self-evolution" but no implementation exists.

## Alternatives

| System | Use When |
|---|---|
| [EvoAgentX](../projects/evoagentx.md) | You want workflow-level evolution (prompt rewriting, structure modification) without environment RL training |
| [DSPy](../projects/dspy.md) | You need [Prompt Optimization](../concepts/prompt-optimization.md) via compilation rather than RL fine-tuning |
| [Voyager](../projects/voyager.md) | Your target is open-world exploration with skill accumulation rather than structured API benchmarks |
| [Darwin Gödel Machine](../projects/darwin-godel-machine.md) | You need self-modifying code that rewrites its own architecture, not just its policy weights |
| [AFlow](../projects/aflow.md) | You want automated workflow optimization without the compute requirements of full RL training |
| [Reflexion](../concepts/reflexion.md) | You need self-improvement via verbal reflection within a single context window, without weight updates |

Use AgentEvolver when you have a well-defined environment with automated evaluation, GPU training infrastructure, and want to produce a fine-tuned model rather than improve inference-time behavior.
