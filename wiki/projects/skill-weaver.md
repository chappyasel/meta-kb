# SkillWeaver

## What It Does

SkillWeaver is a framework for building self-improving LLM agents through automated skill synthesis. Agents encounter tasks, generate reusable Python functions (skills) to solve them, verify those skills against test cases, then compose previously acquired skills into solutions for new tasks. The core bet: a growing verified skill library compounds across tasks rather than each task being solved from scratch.

It appears in the MemEvolve paper's EvolveLab framework as one of 12 baseline memory systems, reimplemented under `EvolveLab/providers/skillweaver.py` and registered in the `PROVIDER_MAPPING` via `MemoryType.SKILLWEAVER`. This positions it within the "encode-store-retrieve-manage" taxonomy that MemEvolve uses to classify memory architectures, meaning SkillWeaver's design choices around how skills are encoded, stored, and retrieved are explicit comparison points against evolved alternatives.

The arXiv paper is [2504.07079](https://arxiv.org/abs/2504.07079), from OSU-NLP-Group.

## Core Mechanism

The system runs three coordinated phases per task:

**Synthesis**: An LLM generates a Python function that solves the current task, given the task description and any relevant skills retrieved from the library. Functions include docstrings, typed signatures, and example invocations that double as test scaffolding.

**Verification**: Generated skills run against automatically constructed test cases. Skills that fail verification are not added to the library. This is the critical quality gate: without it, the library accumulates broken or task-specific functions that degrade retrieval quality. The verification mechanism distinguishes SkillWeaver from naive function-caching approaches.

**Composition**: When a new task arrives, a retrieval step pulls semantically similar skills from the library (likely embedding-based similarity over docstrings or function names). The LLM receives the retrieved skills as context and can call them in its generated solution, building on prior work rather than rewriting.

The skill library maps to EvolveLab's `BaseMemoryProvider` interface via `provide_memory(MemoryRequest) -> MemoryResponse` (retrieval) and `take_in_memory(TrajectoryData) -> (bool, str)` (synthesis and storage after task completion). Each skill persists as a stored artifact, placing SkillWeaver in the E-U-R-G taxonomy at roughly: encode=function+docstring, store=file/vector-index, retrieve=embedding similarity, manage=deduplication/pruning.

This architecture is closest to Voyager (Minecraft agent with a growing code skill library) applied to general LLM agent tasks, with verification added as a robustness layer.

## Key Numbers

The project has **286 GitHub stars** as of the MemEvolve paper's writing (self-reported via README badge; not independently audited). Benchmark figures come from the SkillWeaver paper itself (self-reported). MemEvolve's evaluation treats SkillWeaver as a baseline and reports that evolved memory systems outperform it on GAIA, WebWalkerQA, and xBench, though the margin varies by framework and model. No third-party reproduction of SkillWeaver's reported numbers is cited in available sources.

## Strengths

**Compounding across tasks**: Skills that survive verification become available to all subsequent tasks. On domains with genuine reuse structure (e.g., web scraping, data transformation, API calls), a mature library meaningfully reduces per-task LLM load.

**Quality gate via verification**: Automatically rejecting skills that fail test cases keeps library quality high. This prevents the silent degradation that plagues systems storing unvalidated experience, where retrieved but broken functions mislead the agent.

**Interpretability**: Skills are readable Python functions with docstrings. A human can audit the library, remove bad skills, or manually add domain knowledge. This is a concrete advantage over embedding-based or latent memory systems.

**Composability**: Retrieved skills reduce the solution space the LLM must explore. On tasks requiring multi-step pipelines, composition of verified primitives is more reliable than generating a monolithic solution.

## Critical Limitations

**Fixed architecture ceiling**: SkillWeaver's memory architecture is static. The schema for what a "skill" is (a Python function with a docstring) is set at design time. It cannot decide that a particular task domain would be better served by a lookup table, a retrieval-augmented knowledge base, or a state machine. This is precisely the limitation MemEvolve was built to transcend, and why SkillWeaver sits in EvolveLab as a baseline rather than an end point.

**Concrete failure mode**: Verification tests are generated by the same LLM that generates the skill, or derived from the task that triggered synthesis. A skill that passes its own tests but is subtly wrong for edge cases enters the library and gets retrieved for future tasks. The composition mechanism then propagates the error: a downstream skill built on a flawed primitive inherits the flaw, fails on the new task, and the debugging signal points to the downstream skill rather than the library contamination. Library poisoning is hard to detect without external ground truth.

**Unspoken infrastructure assumption**: The framework assumes tasks arrive in a sequence that allows skill amortization. If each task is from a different domain, retrieval returns low-relevance results, synthesis produces task-specific one-offs, and the library grows without yielding compounding returns. The self-improvement story depends on a task distribution with meaningful overlap, which the system does not verify or adapt to.

## When Not to Use It

Skip SkillWeaver when:

- Tasks are highly heterogeneous with little structural overlap. The library fills with single-use functions, retrieval becomes noisy, and you're paying synthesis overhead with no compounding benefit.
- The target domain requires stateful or long-horizon reasoning where Python functions are too coarse-grained as primitives.
- You need guaranteed correctness. Verified-against-own-tests is not equivalent to correct, and downstream composition amplifies any residual errors.
- Latency is constrained. Each task incurs retrieval, potential synthesis, and verification before execution. Cold-start tasks pay full cost with no library benefit.
- The task environment changes frequently. Skills verified against one API version or data schema silently break when the environment shifts, and the library has no staleness detection.

## Unresolved Questions

The available documentation does not address:

**Library management at scale**: What happens when the library contains thousands of skills? Does retrieval quality degrade? Is there a pruning or consolidation strategy, or does the library grow indefinitely?

**Skill deduplication**: When two similar skills are synthesized for related tasks, does the system merge them, keep both, or prefer the newer version? Duplicate or near-duplicate skills fragment retrieval.

**Verification test quality**: Who generates the verification tests, and how is their coverage assessed? If tests are LLM-generated from the same prompt that generated the skill, coverage may be systematically incomplete in the same ways.

**Cross-domain transfer governance**: No documented policy for which skills are scoped to a task type versus made globally retrievable. A SQL-generation skill retrieved in a context expecting web API calls could confuse the LLM rather than help it.

**Cost at scale**: Full skill synthesis (LLM call) + verification (execution) + embedding + storage happens for every new skill. At high task volumes with frequent novel tasks, this is non-trivial. No published cost analysis.

## Alternatives

| System | Use When |
|--------|----------|
| **ExpeL** | You want experience stored as natural language examples rather than executable code; lower execution risk, lower precision |
| **Agent Workflow Memory** | Tasks follow identifiable workflow patterns; you want inductive generalization over trajectories rather than synthesized functions |
| **MemEvolve** | You want the memory architecture itself to adapt to task demands, not just the memory contents; willing to pay LLM code-generation cost for meta-level optimization |
| **Voyager** | Operating in a structured simulation environment (Minecraft-like) where executable skills map cleanly to environment actions |
| **Vanilla RAG over trajectories** | Simplicity matters more than compounding; you want to retrieve past solutions without the synthesis and verification overhead |

SkillWeaver is the right choice when tasks share a reusable-function structure, you can tolerate imperfect verification, and you want human-auditable skill artifacts. It loses to MemEvolve when the skill-as-Python-function schema itself becomes the bottleneck.
