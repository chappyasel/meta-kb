# BM25

## What It Is

BM25 (Best Matching 25) is a probabilistic term-weighting function from the Okapi family of retrieval models, developed at City University London in the 1990s. Given a query Q and a set of documents, BM25 scores each document by how well its terms match the query, accounting for term frequency saturation and document length.

The core scoring formula for a document D against query Q:

```
score(D, Q) = Σ IDF(qi) * [f(qi, D) * (k1 + 1)] / [f(qi, D) + k1 * (1 - b + b * |D| / avgdl)]
```

Where:
- `f(qi, D)` = frequency of query term `qi` in document D
- `|D|` = document length, `avgdl` = average document length in corpus
- `k1` (default 1.2–2.0): controls term frequency saturation
- `b` (default 0.75): controls length normalization
- `IDF(qi)` = inverse document frequency, penalizing terms that appear everywhere

The key insight is **term frequency saturation**: once a term appears enough times, additional occurrences add diminishing marginal value. A document mentioning "authentication" 20 times is not 20x more relevant than one mentioning it once. `k1` controls where this plateau kicks in.

BM25 is not a model you train. It computes scores deterministically from term statistics over your corpus. That makes it fast, reproducible, and interpretable — you can explain exactly why any document ranked where it did.

## Why It Matters for RAG and Agent Memory

Most production retrieval systems pair BM25 with vector embeddings in [Hybrid Retrieval](../concepts/hybrid-retrieval.md) pipelines. The reason is that each method covers the other's blind spots:

- BM25 matches exact terms. If the query contains a product code, a person's name, a technical acronym, or any string the user typed literally, BM25 finds it. Embeddings can miss this if the term's vector representation diffuses the signal.
- Embeddings match semantic intent. If the query says "login issues" and documents say "authentication failures," embeddings bridge the gap. BM25 finds nothing.

This complementarity makes BM25 the standard first-stage or co-stage retriever in [Retrieval-Augmented Generation](../concepts/rag.md) pipelines. Systems like [Graphiti](../projects/graphiti.md) run BM25 fulltext search in parallel with cosine similarity and graph traversal, then merge results with Reciprocal Rank Fusion. The [Napkin](../projects/napkin.md) project makes the stronger claim that BM25 alone — without any embeddings — achieves 91% on LongMemEval-S on well-structured markdown knowledge bases, outperforming embedding-based RAG.

BM25 also appears in [Semantic Memory](../concepts/semantic-memory.md) and [Episodic Memory](../concepts/episodic-memory.md) implementations. The Mem-Alpha system uses BM25 alongside embedding search as dual retrieval modes for its three-tier memory store, letting agents choose the right tool per query type.

## How It Works: Implementation Details

### Algorithm Steps

1. **Build an inverted index** from your corpus: for each unique term, record which documents contain it and how many times.
2. **Compute IDF scores** for each term: `log((N - df + 0.5) / (df + 0.5) + 1)`, where N is total documents and df is document frequency. High-frequency terms (appearing in 80% of documents) get near-zero IDF. Rare terms get high IDF.
3. **At query time**, tokenize the query, look up each term in the inverted index, compute the BM25 formula, sum across query terms, rank.

### Common Implementations

- **rank_bm25** (Python): Pure-Python, used in Mem-Alpha. `from rank_bm25 import BM25Okapi`. Simple to deploy, no infrastructure dependency.
- **MiniSearch** (JavaScript/TypeScript): Used in Napkin. BM25 with configurable fuzzy matching and field-weighted search (filenames boosted 2x over body text).
- **Elasticsearch / OpenSearch**: Production-grade distributed BM25 via the `match` query. Used widely in enterprise search.
- **Neo4j / FalkorDB native fulltext**: Graph databases include BM25 over node and edge properties — this is what Graphiti uses for its `phi_bm25` search function.
- **SQLite FTS5**: Built into SQLite, BM25-based, zero infrastructure.
- **Lucene** (Java): BM25 is Lucene's default similarity since version 6.0, underlies Elasticsearch and Solr.

### Key Parameters

`k1` and `b` are the tunable parameters. The defaults work for most cases:

| Setting | k1 | b | Use Case |
|---------|-----|---|----------|
| Default | 1.2 | 0.75 | General text |
| Higher k1 (2.0) | 2.0 | 0.75 | Longer documents where term frequency matters more |
| Lower b (0.5) | 1.2 | 0.5 | When document length varies widely and shouldn't dominate |
| b=0 | 1.2 | 0.0 | Disables length normalization entirely |

### Tokenization Choices

BM25 scores depend entirely on what tokens you produce. Decisions that matter:

- **Stemming** (porter, snowball): "running" and "runs" both match "run". Improves recall, reduces precision.
- **Lowercasing**: standard. "Authentication" and "authentication" should match.
- **Stopword removal**: removes "the", "is", "at". Reduces index size, but can hurt phrase queries.
- **Field weighting**: boost title/heading terms over body terms. Napkin weights headings 3x over body.
- **n-gram tokenization**: include bigrams ("machine learning") alongside unigrams to match multi-word concepts.

## Strengths

**Exact-match reliability**: Product codes, names, identifiers, technical terms, and any string the user typed will be found if they appear in documents. This is BM25's strongest suit and why it's irreplaceable in hybrid systems.

**Zero preprocessing**: No embedding model, no GPU, no vector index build time. Feed BM25 a corpus, get scores back immediately. When a document changes, update the inverted index entry for affected terms — incremental updates are fast. Embedding pipelines require recomputing vectors for changed content.

**Interpretability**: You can explain every score. "This document ranked #1 because 'authentication' appears 4 times (TF score 0.82) and appears in only 3% of documents (IDF score 3.47)." No black box.

**Computational cost**: Lookups in an inverted index are fast. For corpora of millions of documents, BM25 query time is milliseconds. Embedding cosine similarity at the same scale requires approximate nearest-neighbor infrastructure (HNSW, IVF).

**No semantic drift**: Embeddings trained on one domain can produce misleading similarity scores in another. BM25 scores derive entirely from term statistics in your specific corpus — no model-domain mismatch.

## Limitations

### Vocabulary Gap (The Core Failure Mode)

BM25 scores zero for documents that don't share any query terms. A query for "login issues" finds no results in a document about "authentication failures" unless both terms co-occur somewhere. This is the fundamental limitation that makes pure BM25 insufficient for most semantic retrieval tasks.

This failure is not theoretical. If your knowledge base uses consistent internal vocabulary, BM25 works well. If users query with different words than your authors used, BM25 misses. Domain-specific corpora (medical notes, legal filings, code documentation) are especially vulnerable because terminology is specialized and users may not know the exact terms authors chose.

### Unspoken Infrastructure Assumption

BM25 implementations make different assumptions about corpus size and update patterns:

- `rank_bm25` (Python) loads the entire index into memory. At 100K+ documents with long content, RAM consumption becomes a concern.
- Native database BM25 (Neo4j, Elasticsearch) handles scale but requires the database to be running and configured with fulltext indices.
- MiniSearch serializes the index to JSON for caching, which breaks down with large corpora.

No BM25 library handles corpus changes elegantly for streaming ingestion at high volume. You either batch-rebuild the index or use a system with a native incremental update path (Elasticsearch handles this; rank_bm25 does not).

### Phrase and Proximity Blindness

Standard BM25 has no concept of term proximity. "John Smith" and "Smith John" score identically for a query for "John Smith." Phrase matching requires either exact-match handling or a different retrieval model (BM25F, span queries). Most BM25 implementations in agent memory systems ignore this.

### No Learning

BM25 cannot incorporate feedback. Click data, thumbs-up ratings, or user corrections don't update BM25 scores. The IDF values update when documents enter or leave the corpus, but the relevance function itself doesn't adapt. This matters for agent memory systems where retrieval quality should improve over time — learned rerankers or embedding fine-tuning can improve, BM25 cannot.

## When NOT to Use BM25 Alone

**Synonymy-heavy domains**: Medical, legal, and scientific text often use multiple terms for the same concept. "MI", "myocardial infarction", "heart attack" are the same; BM25 treats them as different. Add embeddings.

**Short documents with few terms**: Very short memory entries (< 20 words) give BM25 little signal. The IDF denominator may not stabilize, and term frequency saturation curves give misleading scores. Embeddings handle short texts better.

**Cross-lingual retrieval**: BM25 requires term overlap. A query in English against documents in French finds nothing. Multilingual embeddings handle this; BM25 cannot.

**Fuzzy requirements over strict recall**: If your system should surface "relevant neighborhood" rather than "documents containing these exact words," embeddings are the right primary retrieval mechanism. BM25 as a re-ranker or co-retriever still adds value.

**When you have well-trained dense retrievers on your domain**: Dense retrieval fine-tuned on domain-specific data can outperform BM25 on that domain. Fine-tuned embeddings absorb the vocabulary of your corpus, closing the synonym gap. At that point, BM25 becomes a fallback for exact-match edge cases rather than a co-equal retrieval signal.

## Integration Patterns

### Hybrid Retrieval with RRF

The standard production pattern: run BM25 and embedding search in parallel, merge with Reciprocal Rank Fusion.

```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```

Where `k` is a constant (typically 60) and `rank_i(d)` is document d's rank in result set i. RRF is robust to score scale mismatches between BM25 (arbitrary units) and embedding similarity (typically -1 to 1).

Graphiti's search layer runs `cosine_similarity`, `bm25`, and `bfs` (graph traversal) in parallel, then applies RRF or one of several other reranking strategies. This three-signal approach — word similarity, semantic similarity, contextual/graph similarity — targets complementary retrieval dimensions.

### BM25 for Pre-filtering

In systems with expensive rerankers (cross-encoders, LLM-based reranking), BM25 filters the corpus to a candidate set cheaply, then the expensive model scores only the candidates. Retrieve 100 with BM25, rerank the top 100 with a cross-encoder, return the top 5. This is the standard retrieve-then-rerank pattern.

### Field-Weighted BM25

Different document sections deserve different weights. Napkin's approach: headings 3x, filenames/titles 2x, body 1x. Elasticsearch's BM25F (Field extension) formalizes this. The intuition: if a document is titled "Authentication" it is more about authentication than a document that mentions the word once in a footnote.

### BM25 + Backlinks (Napkin Pattern)

Napkin combines BM25 with a link-graph signal:

```
composite = BM25_score + backlink_count * 0.5 + recency_normalized * 1.0
```

The backlink count is effectively a simplified PageRank — documents that many others link to are treated as more authoritative. This is specific to knowledge bases with explicit wikilink structure, but the pattern generalizes: any graph signal that measures document authority can be combined with BM25 scores.

### Source of Truth: Markdown, Not BM25 Index

A pattern appearing across multiple projects: treat markdown files as the source of truth and BM25 indexes as derived, rebuildable artifacts. When files change, invalidate the BM25 index (via content hash or file modification time). Never store critical information only in the BM25 index. This keeps the knowledge base portable and auditable.

## Relationship to Related Concepts

BM25 is one retrieval mechanism within the broader [Hybrid Retrieval](../concepts/hybrid-retrieval.md) architecture used in [Retrieval-Augmented Generation](../concepts/rag.md) systems. It pairs with [Vector Database](../concepts/vector-database.md) infrastructure for the semantic half of hybrid search.

In [Semantic Memory](../concepts/semantic-memory.md) and [Episodic Memory](../concepts/episodic-memory.md) implementations, BM25 handles exact-term lookup while embeddings handle conceptual similarity. Systems like [Graphiti](../projects/graphiti.md) run both in parallel. [Mem0](../projects/mem0.md) uses BM25 for keyword search over stored memory entries.

The Meta-Harness research found that optimized BM25 routing (subject-specific BM25 policies per query type) outperformed dense retrieval alone on IMO-level math problems, demonstrating that the simple scoring function can outperform learned embeddings when the query characteristics are understood and the retrieval policy is tuned accordingly.

## Unresolved Questions

**Optimal corpus granularity**: Should you index full documents, paragraphs, or sentences? BM25 scores depend on document length normalization — finer granularity gives more precise length normalization but fragments context. There is no consensus on optimal chunking strategy for BM25 specifically.

**Parameter tuning without labeled data**: k1 and b defaults work generally but optimal values depend on corpus characteristics. Tuning requires a labeled evaluation set. Most agent memory systems use defaults without validation on their specific corpus.

**Index freshness for streaming ingestion**: None of the lightweight Python BM25 libraries (rank_bm25, BM25s) document how to handle high-volume streaming updates without full index rebuilds. Production systems typically use Elasticsearch or similar for this.

## Alternatives: Selection Guidance

| When | Use |
|------|-----|
| Need semantic synonym matching without exact terms | Embedding search (dense retrieval) |
| Need both exact and semantic matching | Hybrid BM25 + embeddings with RRF |
| Structured knowledge base with consistent vocabulary and zero infrastructure budget | BM25 alone (Napkin demonstrates this is viable) |
| Production scale, multiple users, managed infrastructure | Elasticsearch or OpenSearch with BM25 queries |
| Graph-structured data with relationship traversal | BM25 + graph traversal (Graphiti pattern) |
| Domain-specific corpus with training data available | Fine-tuned dense retriever, BM25 as fallback |
