# BM25

## What It Is

BM25 (Best Match 25) is a bag-of-words retrieval function developed through the Okapi research project at City University London in the 1990s. Given a query with terms, it scores every document in a corpus and ranks them by relevance. The algorithm sits in the "sparse retrieval" family: it represents documents and queries as sparse vectors over a vocabulary, where most values are zero, and computes relevance from term overlap alone.

BM25 is the default ranking function in Elasticsearch, Solr, and Tantivy. It underlies search in most enterprise document systems. In the LLM agent space, it appears as a first-stage retriever in [Hybrid Search](../concepts/hybrid-search.md), as a component of [Retrieval-Augmented Generation](../concepts/retrieval-augmented-generation.md) pipelines, and — more surprisingly — as a standalone retriever competitive with embedding-based approaches on structured knowledge bases.

## How It Works

### The Scoring Formula

For a query Q containing terms q₁...qₙ and a document D:

```
Score(D, Q) = Σᵢ IDF(qᵢ) · (f(qᵢ, D) · (k₁ + 1)) / (f(qᵢ, D) + k₁ · (1 - b + b · |D|/avgdl))
```

Where:
- `f(qᵢ, D)` is the term frequency of qᵢ in document D
- `|D|` is document length in words
- `avgdl` is the average document length across the corpus
- `k₁` controls term frequency saturation (typically 1.2–2.0)
- `b` controls length normalization (typically 0.75)
- `IDF(qᵢ) = log((N - n(qᵢ) + 0.5) / (n(qᵢ) + 0.5) + 1)` where N is total documents and n(qᵢ) is documents containing the term

Three design choices distinguish BM25 from raw TF-IDF:

**Term frequency saturation.** Raw TF-IDF scores grow linearly with term count: a document mentioning "authentication" 100 times scores 100× higher than one mentioning it once, which is rarely appropriate. BM25's k₁ parameter creates a saturation curve — beyond a threshold, additional term occurrences contribute diminishing returns. With k₁ = 1.5, moving from 1 to 2 occurrences nearly doubles the score; moving from 10 to 11 adds almost nothing.

**Length normalization.** Longer documents naturally accumulate more term matches. The b parameter controls how aggressively to penalize length. At b = 0, no length normalization; at b = 1, full normalization relative to corpus average. The typical b = 0.75 is a partial normalization — short, dense documents still benefit slightly over long documents that match only by volume.

**IDF smoothing.** The +0.5 smoothing prevents division-by-zero and reduces the impact of very rare terms that appear in exactly one document.

### Index Structure

BM25 operates on an inverted index: for each vocabulary term, a posting list records which documents contain that term and at what frequency. At query time, the system retrieves posting lists for each query term, computes the score formula for each matching document, and returns a sorted ranking.

Building this index requires:
1. Tokenization (splitting text into terms)
2. Stopword removal (dropping "the", "a", "is")
3. Optional stemming or lemmatization ("running" → "run")
4. Computing document lengths and corpus statistics (N, avgdl, per-term document frequency)

After construction, the index is a read-heavy data structure. BM25 scoring touches only documents that contain at least one query term — compared to dense retrieval, which must compute similarity against every document or use approximate nearest-neighbor structures.

### BM25 in Practice: Four Implementations Observed

**MiniSearch (JavaScript, used in Napkin):** A lightweight in-memory BM25 library, serializable to JSON. Napkin uses MiniSearch with custom field weights — `basename` (filename) boosted 2× over `content` — plus a composite ranking signal:

```
composite = BM25_score + backlink_count * 0.5 + recency_normalized * 1.0
```

This augments vocabulary-based matching with graph structure (Obsidian wiki-links) and recency, addressing two of BM25's structural weaknesses. The [Napkin source analysis](../raw/deep/repos/michaelliv-napkin.md) notes that scores are hidden from agent output to prevent LLM anchoring on numeric confidence values — the sort order reaches the model, but not the numbers.

**Tantivy (Rust, used in Memvid):** A full-featured Rust BM25 engine with Lucene-compatible behavior. Memvid embeds Tantivy's segment files inside its `.mv2` binary format — extracting them to a temp directory on open, reserializing on commit. This preserves the single-file portability guarantee while using a production-grade BM25 implementation. [Source](../raw/deep/repos/memvid-memvid.md) confirms Tantivy is the primary BM25 backend, with a custom hand-rolled BM25 (`src/lex.rs`) as fallback.

**rank_bm25 (Python, used in Mem-alpha):** The academic memory system [Mem-alpha](../raw/deep/repos/wangyu-ustc-mem-alpha.md) uses the `rank_bm25` Python library for keyword-based semantic and episodic memory search, alongside text-embedding-3-small for semantic search. The system lets agents (and the QA evaluator) choose between BM25 and embedding search per query — a design that acknowledges both approaches have query-type-specific strengths.

**Elasticsearch / Lucene (widespread):** Most production RAG deployments layer BM25 through Elasticsearch, with dense vector search added as a separate field or via a hybrid scoring plugin. The query-time combination typically uses Reciprocal Rank Fusion (RRF) to merge BM25 and vector rankings without requiring a shared score scale.

**Subject-specific BM25 routing (discovered by Meta-Harness):** The [Meta-Harness](../raw/deep/papers/lee-meta-harness-end-to-end-optimization-of-model-har.md) optimization system automatically discovered a specialized BM25 routing strategy for math retrieval:
```
Combinatorics: BM25@20, deduplicate to 8, rerank, keep top 3
Geometry: 1 hard reference + 2 BM25 neighbors
Number Theory: BM25@12, rerank with technique-early bonus, keep 3
Default: BM25@10, adaptive count based on score concentration
```
This is noteworthy: a learned optimizer, given the freedom to invent any retrieval strategy, chose to keep BM25 as the core mechanism and specialize parameters per subject domain rather than replacing it with dense retrieval.

## Strengths

**Zero inference cost.** Building a BM25 index requires no neural network. Querying requires no GPU. The entire retrieval pipeline runs on CPU with minimal memory. For local-first systems or edge deployment, this is the decisive advantage over embedding-based approaches.

**Exact keyword matching.** When a user queries a specific function name, error code, product ID, or proper noun, BM25 finds exact matches reliably. Embedding models sometimes "helpfully" retrieve semantically similar documents that use different terminology, introducing false positives that frustrate users who wanted the specific term.

**Interpretable ranking.** Every score has a traceable explanation: this document scored higher because term X appears 3 times and is rare in the corpus. Embedding cosine similarity provides no such explanation.

**Lexical freshness.** Adding a new document to a BM25 index requires updating posting lists and recalculating IDF — fast operations. Embedding-based indexes require running the new document through an embedding model. BM25 indexes reflect corpus changes with lower latency.

**Competitive on structured knowledge.** The Napkin benchmark results challenge the assumption that embeddings are necessary for high-recall retrieval. On LongMemEval-S (40 sessions per question), BM25 with progressive disclosure achieved 91% accuracy versus 86% for the best prior embedding-based system. This result holds specifically because the knowledge base was structured markdown with consistent terminology — the scenario where vocabulary overlap is high.

## Critical Limitations

**The vocabulary gap.** BM25 requires term overlap between query and document. "Authentication" does not match "login". "Cardiovascular disease" does not match "heart attack" unless both terms appear together. This is BM25's fundamental constraint, and there is no workaround within the algorithm itself.

Concrete failure mode: an agent asks "how do we handle login failures?" and the relevant documentation uses "authentication errors" and "credential validation failures" throughout. BM25 returns no matches. An embedding-based system retrieves the correct documents because the semantic representations are close in vector space.

**Infrastructure assumption: vocabulary stability.** BM25 performs best when the corpus uses consistent terminology. Internal documentation, code comments, and structured notes tend to satisfy this assumption. User-generated content, multilingual corpora, or rapidly evolving domains with terminology drift do not. Systems built on BM25 assume that whoever writes documents and whoever queries them share a vocabulary.

**No contextual disambiguation.** "Python" the programming language and "python" the snake receive identical BM25 treatment. The algorithm has no model of word sense. Embedding models encode context and partially resolve this.

**Parameter sensitivity at scale.** The k₁ and b hyperparameters interact with corpus statistics. Values tuned on a small corpus may perform poorly on a large one, or vice versa. Most systems use defaults (k₁ ≈ 1.5, b ≈ 0.75) without corpus-specific tuning, which works acceptably but leaves performance on the table.

## When NOT to Use It

**When queries and documents use different vocabularies.** Customer support chat (informal language) over technical documentation (formal language), cross-language retrieval, or domains where jargon varies by community. Embedding models handle vocabulary gap; BM25 does not.

**When semantic meaning matters more than lexical overlap.** "What should I do if I feel overwhelmed?" deserves to retrieve documents about stress management even if they never use the word "overwhelmed." Therapeutic, educational, or exploratory retrieval tasks require semantic understanding.

**When the corpus is unstructured and noisy.** OCR output, scraped web content, or mixed-format files produce messy tokenization. BM25 scores on raw tokens — garbage in, garbage out. Embedding models are somewhat more robust to surface-level noise.

**When you need zero cold-start latency on first query.** MiniSearch and similar in-memory BM25 implementations rebuild the full index on the first query if the cache is cold (as in Napkin's fingerprint-based cache). For vaults with 10K+ files, this adds noticeable latency on session start.

## Hybrid Search: BM25 + Dense Retrieval

In production RAG systems, BM25 rarely operates alone. [Hybrid Search](../concepts/hybrid-search.md) combines BM25's lexical precision with dense retrieval's semantic recall, typically merged via Reciprocal Rank Fusion:

```
RRF_score(d) = Σ 1/(k + rank_in_list(d))
```

where k = 60 is the standard smoothing constant. RRF avoids the score normalization problem — BM25 scores and cosine similarities live on different scales and cannot be directly summed. RRF uses only rank positions, which are scale-invariant.

The practical effect: queries with exact terminology get BM25-dominated results (high precision), queries with paraphrases or synonyms get embedding-dominated results (high recall), and most queries benefit from both. [Memvid's source](../raw/deep/repos/memvid-memvid.md) implements this pattern for its `ask()` endpoint, using RRF with k=60 to merge Tantivy BM25 results with HNSW vector results.

## BM25 in the Agent Memory Stack

BM25 appears at multiple layers of agent knowledge infrastructure:

**As a routing signal in [Retrieval-Augmented Generation](../concepts/retrieval-augmented-generation.md).** BM25 runs cheaply on every query to filter candidate documents before expensive reranking or embedding-based retrieval. This two-stage architecture (cheap BM25 filter → expensive reranker) dominates production RAG.

**As a navigation primitive for [Context Management](../concepts/context-management.md).** Napkin's progressive disclosure architecture uses BM25 to navigate a knowledge vault without loading everything into context. The agent queries BM25, reads snippet previews, then fetches full files only for relevant results — a three-level protocol that respects context window budgets.

**As a component of [Agent Memory](../concepts/agent-memory.md) retrieval.** Mem-alpha provides both BM25 and embedding search to the QA agent, letting it choose per query. The memory agent (trained via RL) learns which memory type and search method to use for each query pattern.

**As a skill-selection primitive.** [Memento](../projects/memento.md) uses BM25 keyword search (`LocalFileRecall`) as one of three parallel recall strategies for selecting which skills to activate for a given task — the cheapest recall strategy in the multi-recall fusion.

## Alternatives and Selection Guidance

**Use BM25 when:** the corpus has consistent terminology, queries use domain-specific terms that must match exactly, you need zero infrastructure (no GPU, no vector store), or you are building a local-first system with a cold-start latency budget.

**Use dense retrieval (embeddings + [Vector Database](../concepts/vector-database.md)) when:** vocabulary gap is a real problem, semantic similarity matters more than lexical overlap, or the corpus is diverse and unstructured.

**Use hybrid BM25 + dense when:** you need both precision (exact match) and recall (semantic match), which is most production RAG deployments. The overhead is manageable: BM25 index + embedding index in parallel, merged via RRF at query time.

**Use [Semantic Search](../concepts/semantic-search.md) exclusively when:** the domain has high synonym density, the corpus is multilingual, or the queries are exploratory ("find things related to X") rather than lookup-oriented ("find the document about X").

**Consider BM25-only + progressive disclosure (Napkin pattern) when:** the knowledge base is structured markdown, terminology is controlled, and the goal is agent memory rather than open-domain search. The Napkin benchmarks suggest this configuration outperforms embedding-based RAG on structured, well-maintained knowledge bases — and requires no vector store infrastructure.

## Unresolved Questions

**Optimal parameters for agent memory corpora.** Most systems use default k₁ and b values. Agent memory corpora have distinctive properties — short notes, consistent terminology, high link density — that may warrant different tuning. No published work establishes BM25 parameters specifically for agent memory.

**Vocabulary gap threshold.** At what synonym density does BM25 fail badly enough to require embedding augmentation? The Napkin results suggest structured documentation stays below this threshold; user conversations likely exceed it. The boundary has not been formally characterized.

**Interaction with chunking strategy.** BM25 scores depend on document length (via b) and term frequency relative to document length. Chunking strategies that produce variable-length segments will interact with these parameters in non-obvious ways. The Napkin benchmark insight — per-round notes outperform full-session notes for BM25 — suggests chunk granularity matters significantly, but optimal granularity is task-dependent and understudied.
