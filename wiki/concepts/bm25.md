# BM25

## What It Is

BM25 (Best Matching 25) is a sparse retrieval ranking function from the probabilistic relevance framework, developed by Stephen Robertson and colleagues at City University London in the 1990s. It scores documents against a query by combining term frequency saturation, document length normalization, and inverse document frequency. The "25" refers to the 25th iteration of the underlying probabilistic model, not a parameter value.

BM25 is the dominant classical baseline in information retrieval and appears in virtually every Retrieval-Augmented Generation pipeline, either as the primary retriever or as the sparse component in hybrid search.

## How It Works

For a query Q with terms q₁...qₙ and a document D, BM25 scores:

```
Score(D, Q) = Σ IDF(qᵢ) · [f(qᵢ, D) · (k₁ + 1)] / [f(qᵢ, D) + k₁ · (1 - b + b · |D|/avgdl)]
```

Where:
- `f(qᵢ, D)` is raw term frequency of term qᵢ in document D
- `|D|` is document length, `avgdl` is average document length in the corpus
- `k₁` controls term frequency saturation (typically 1.2–2.0)
- `b` controls length normalization (typically 0.75)
- `IDF(qᵢ) = log((N - n(qᵢ) + 0.5) / (n(qᵢ) + 0.5) + 1)`, where N is corpus size and n(qᵢ) is document frequency of the term

Three design choices embedded in this formula matter:

**Term frequency saturation.** Raw TF grows linearly; BM25 TF grows but hits a ceiling. A document mentioning "database" 100 times does not score 100x higher than one mentioning it once. The parameter k₁ controls how fast this ceiling is reached.

**Length normalization.** Long documents naturally accumulate more term occurrences. BM25 penalizes length via the `b` parameter, scaling against corpus-average document length. Setting b=0 removes this normalization entirely.

**IDF weighting.** Rare terms score higher. Terms appearing in almost every document (like "the") contribute near zero, which is a stronger penalty than classic TF-IDF provides due to the +0.5 smoothing constants.

The result is an inverted index lookup: at query time, retrieve posting lists for each query term, compute per-document BM25 scores, and return the top-k. No matrix multiplications, no GPU, no network calls to an embedding API.

## Role in RAG Systems

BM25 handles what dense retrieval cannot: exact matches. Product SKUs, error codes, contract clause numbers, legal citations, proper nouns, version strings — these fail semantic search because embeddings represent meaning, not orthography. A query for "errno 13" finds documents containing "errno 13" via BM25; an embedding model may return documents about "permission denied" that do not contain the string.

In production hybrid pipelines, BM25 and vector search run in parallel, and their result lists are merged via Reciprocal Rank Fusion (RRF) or learned score interpolation before optional cross-encoder reranking. The [DEV Community article on RAG production failures](../../raw/articles/dev-community-why-most-rag-systems-fail-in-production-and-how-t.md) explicitly names "using only vector search" as a common production mistake.

Benchmarks from [napkin](../../raw/repos/michaelliv-napkin.md) (264 GitHub stars, MIT) demonstrate BM25 on markdown files reaching 83% on LongMemEval-M (500 sessions), outperforming the prior best system at 72% and GPT-4o with full context which the benchmark does not report. This is self-reported by the napkin project and has not been independently validated.

The [hipocampus](../../raw/repos/kevin-hs-sohn-hipocampus.md) benchmark (MemAware, 900 implicit context questions across 3 months of agent conversation history) shows BM25 search scoring 2.8% overall versus 0.8% for no memory at all, and 3.4% for BM25 + vector combined. These numbers look poor in isolation, but the benchmark tests implicit context retrieval — the agent must surface information the user never mentioned — which structurally disadvantages any search mechanism. BM25 alone scores 4.7% on easy questions and 2.0% on hard, confirming it degrades on cross-domain queries with zero keyword overlap. Self-reported benchmark.

## Strengths

**No infrastructure.** BM25 runs on an inverted index, which every major database already provides (Elasticsearch, PostgreSQL with `tsvector`, SQLite FTS5, Solr). No embedding model, no GPU, no vector database subscription.

**Exact match precision.** Identifiers, codes, names, and domain-specific terminology retrieve correctly where semantic similarity would return wrong-but-related documents.

**Interpretable scores.** You can see why a document ranked high: which terms matched, how often, against what IDF weight. Debugging a BM25 failure is straightforward.

**Speed at scale.** Inverted index lookups are O(|query terms| · |posting list|), which remains fast on corpora of millions of documents without approximate nearest neighbor approximations.

**No out-of-vocabulary problem.** New terms added to documents are immediately searchable. Embedding models require re-encoding.

## Limitations

**Vocabulary mismatch is the core failure mode.** A user queries "permission denied" and the relevant document says "errno 13 EACCES." BM25 scores this near zero. Synonyms, paraphrases, domain-specific abbreviations, and cross-language content all break BM25 retrieval silently. This is not a tuning problem; it is structural.

**Unspoken infrastructure assumption.** BM25 assumes your documents are already tokenized in a way that matches user query tokenization. In multilingual corpora or technical domains with compound words, this is not trivial. Elasticsearch uses language-specific analyzers; SQLite FTS5 uses a simpler porter stemmer. A mismatch between how "unhelpfulness" gets stemmed at index time versus query time drops recall with no error message.

**No contextual understanding.** BM25 treats each term independently. "Bank" in "river bank" and "bank account" are the same token. Polysemy causes false positives at high recall cutoffs.

**Sparse representation.** The vector BM25 implicitly produces has dimensionality equal to vocabulary size, almost entirely zeros. This is fine for exact retrieval but cannot participate in geometric operations like clustering or similarity to other BM25 vectors.

## When Not to Use BM25 Alone

- When users phrase queries differently from how documents are written (customer-facing search over technical documentation)
- When the corpus is multilingual and queries are monolingual
- When retrieval quality matters more than infrastructure simplicity and you can afford embedding costs
- When queries are short and vague with no distinctive keywords

In all these cases, add dense retrieval and merge results. BM25 alone as the sole retriever in a production RAG pipeline handling natural language questions from non-expert users will underperform.

## Unresolved Questions

**Score normalization across corpora.** BM25 scores are not normalized and are not comparable across different corpora. Merging BM25 and vector scores requires calibration; the standard approach (RRF) sidesteps calibration by using only rank positions, discarding score magnitude entirely. Whether this is the right tradeoff for your retrieval mix is rarely analyzed in individual deployments.

**Parameter sensitivity.** k₁ and b defaults (1.2, 0.75) were derived from TREC experiments on news corpora. On short technical documents, code comments, or conversation logs, optimal parameters differ. Few production deployments tune these, and documentation rarely surfaces this.

**What happens at very high query term counts.** Multi-sentence or paragraph-length queries produce many posting list lookups. BM25 implementations vary in how they handle this: some truncate to top-k terms, some score all. Behavior under long-query conditions is implementation-specific and often undocumented.

## Alternatives and Selection Guidance

| Approach | Use When |
|---|---|
| BM25 alone | Corpus has structured identifiers; budget/latency prohibits embeddings; initial prototype |
| Dense retrieval alone | Queries and documents are semantically rich; no exact-match requirements; corpus is domain-stable |
| BM25 + dense (hybrid) | Production RAG serving real users; mixed query types; when you can afford both retrieval paths |
| Learned sparse (SPLADE, TILDEv2) | You want exact-match behavior with better recall; willing to run a small model at index time |
| Cross-encoder reranking | Top-k precision matters more than recall; BM25 or dense retrieval provides acceptable recall but noisy ranking |
