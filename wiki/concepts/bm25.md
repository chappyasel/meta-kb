# BM25

## What It Is

BM25 (Best Match 25) is a bag-of-words ranking function from the Okapi BM25 family, developed at City University London in the 1990s. It scores documents against a query by combining term frequency, inverse document frequency, and document length normalization. Despite being decades old, BM25 remains the standard baseline for text retrieval and appears as a core component in nearly every [Retrieval-Augmented Generation](../concepts/retrieval-augmented-generation.md) pipeline, often alongside semantic vector search in [Hybrid Search](../concepts/hybrid-search.md) systems.

The "25" refers to the 25th iteration in a series of experiments — not a version number. The final formulation became BM25F in the early 2000s, extending the original to handle structured documents with weighted fields.

## The Scoring Formula

For a query Q containing terms t₁...tₙ and document D:

```
score(D, Q) = Σ IDF(tᵢ) · [f(tᵢ, D) · (k₁ + 1)] / [f(tᵢ, D) + k₁ · (1 - b + b · |D|/avgdl)]
```

Where:
- **f(tᵢ, D)** — term frequency of tᵢ in D
- **|D|** — document length in tokens
- **avgdl** — average document length across the corpus
- **k₁** — term saturation parameter (typically 1.2–2.0); controls how much repeated terms boost the score
- **b** — length normalization parameter (typically 0.75); 0 means no normalization, 1 means full normalization
- **IDF(tᵢ)** — log((N - n(tᵢ) + 0.5) / (n(tᵢ) + 0.5) + 1) where N is total documents and n(tᵢ) is documents containing tᵢ

The key improvement over TF-IDF is the saturation term in the numerator: as f(tᵢ, D) grows, the score increases but with diminishing returns, preventing a document that repeats a query term 100 times from overwhelming a document with better overall coverage.

## How It Works

**Indexing phase:** Text is tokenized, normalized (lowercasing, stemming/lemmatization optionally), and organized into an inverted index: for each term, a posting list records which documents contain it and how many times. Document lengths are recorded for the avgdl calculation.

**Query phase:** For each query term, the posting list is retrieved. The BM25 formula is applied to each candidate document, scores are summed across terms, and results are ranked. This is fundamentally a sparse operation — only documents containing at least one query term get a nonzero score.

**Implementations:** BM25 runs everywhere:
- **Elasticsearch / OpenSearch** — built-in default similarity since Elasticsearch 5.0, replacing TF-IDF
- **Lucene** — the underlying engine for Elasticsearch; `BM25Similarity` class
- **rank_bm25** (Python) — lightweight pure-Python implementation, used in Mem-alpha's memory system [Source](../raw/deep/repos/wangyu-ustc-mem-alpha.md)
- **MiniSearch** (JavaScript) — BM25-based search engine, used in Napkin for vault search [Source](../raw/deep/repos/michaelliv-napkin.md)
- **Tantivy** (Rust) — BM25 full-text engine embedded in Memvid's `.mv2` format [Source](../raw/deep/repos/memvid-memvid.md)
- **Whoosh, Solr, Vespa** — various ecosystems

Parameter defaults (k₁=1.2, b=0.75) work acceptably across most domains. Domain-specific tuning helps: legal and biomedical text benefits from lower b (longer documents are intrinsically more informative, not just verbose).

## Why It Persists

BM25 has survived four decades of research for concrete reasons:

**Zero infrastructure.** No GPU, no embedding model, no vector store. An inverted index is cheap to build, fits on disk, and queries in milliseconds at any scale. Napkin's architecture explicitly credits this: BM25 with progressive disclosure outperforms embedding-based RAG on LongMemEval-S (91% vs 86%) with no preprocessing and no vector database. [Source](../raw/deep/repos/michaelliv-napkin.md)

**Exact keyword matching.** When a user searches for a product SKU, error code, or proper noun, BM25 finds exact matches reliably. Semantic search via embeddings maps "SKU-8734A" into a high-dimensional space where its nearest neighbors are semantically similar but often wrong.

**Interpretability.** Every match explains itself: these terms appeared in these documents with these frequencies. Debugging a BM25 result takes seconds. Debugging a cosine similarity result requires inspecting the embedding model's training data.

**Speed.** At production scale (billions of documents), BM25 over an inverted index is faster than approximate nearest-neighbor search over dense vectors. Elasticsearch serves BM25 queries at millisecond latency across multi-terabyte indexes.

**Composability.** BM25 scores fuse cleanly with other signals. The composite scoring in Napkin's search (`composite = BM25_score + backlink_count * 0.5 + recency_normalized * 1.0`) works because BM25 produces meaningful numeric scores that can be weighted. [Source](../raw/deep/repos/michaelliv-napkin.md) Meta-Harness discovers a subject-specific math retrieval strategy built entirely on BM25 routing (BM25@20 for combinatorics, BM25@12 for number theory) that outperforms naive dense retrieval. [Source](../raw/deep/repos/lee-meta-harness-end-to-end-optimization-of-model-har.md)

## Role in Agent Infrastructure

In the agent/LLM context, BM25 appears in four patterns:

**Pattern 1: Hybrid retrieval.** BM25 retrieves keyword-matched candidates; a semantic embedding model retrieves semantically similar candidates; Reciprocal Rank Fusion (RRF) or a reranker merges the two lists. Memvid implements exactly this — Tantivy BM25 plus HNSW vector search, with RRF fusion at k=60. [Source](../raw/deep/repos/memvid-memvid.md) The combination handles both the "exact term" and "same concept, different words" cases.

**Pattern 2: Skill retrieval.** Memento-Skills uses BM25 (`LocalFileRecall`) as one of three parallel skill retrieval strategies, alongside vector search and remote catalogue search, with local-first priority merging. [Source](../raw/deep/repos/memento-teams-memento-skills.md)

**Pattern 3: Memory search.** Mem-alpha's three-tier memory system (core/semantic/episodic) supports both BM25 and embedding search modes, letting the QA agent choose the appropriate strategy per query. [Source](../raw/deep/repos/wangyu-ustc-mem-alpha.md) Napkin's MiniSearch index is BM25-only, with the backlink count acting as a graph-based reranker.

**Pattern 4: Autonomous harness optimization.** Meta-Harness discovers BM25 routing strategies without being told to use BM25 — the coding agent finds that subject-specific BM25 configurations outperform uniform dense retrieval for math reasoning. [Source](../raw/deep/repos/lee-meta-harness-end-to-end-optimization-of-model-har.md)

## Strengths

**Keyword precision.** Outperforms semantic search for queries with specific technical terms, proper nouns, codes, and identifiers where exact match matters.

**Out-of-vocabulary handling.** BM25 does not care whether a term appears in any training corpus. A new product name or a novel acronym is indexed and retrieved identically to any other term. Embedding models often fail on truly novel terms that fall outside their training distribution.

**No preprocessing lag.** A new document added to a BM25 index is immediately searchable. Embedding pipelines introduce latency between ingestion and retrieval availability.

**Calibrated signals.** BM25 scores are interpretable enough to weight against other signals (recency, backlinks, graph degree) without requiring normalization tricks.

## Critical Limitations

**Vocabulary mismatch.** This is BM25's defining weakness. If a document uses "authentication" and a query asks about "login", BM25 returns nothing useful unless both terms co-occur. Napkin's documentation acknowledges this directly: synonym matching requires embeddings or careful query expansion. Mem-alpha compensates by offering both BM25 and embedding search as alternatives within the same system.

**Unspoken infrastructure assumption: consistent tokenization.** BM25 results depend critically on how text is tokenized before indexing versus at query time. A system that indexes with a standard English tokenizer but queries with Unicode-unaware splitting will silently produce poor results on non-ASCII content. Chinese, Japanese, and Korean text require character-based or specialized tokenizers; the default k₁/b parameters are calibrated on English corpora. Deploying BM25 on multilingual content without matching tokenizers is a common failure mode that produces no visible error — only degraded recall.

## When NOT to Use BM25 Alone

**Semantic similarity tasks.** "Find documents about emotional support for grieving people" will not match documents that use "bereavement counseling" or "grief therapy" unless those exact words appear in the query. Pure BM25 fails on paraphrase and synonym-heavy retrieval tasks.

**Short documents with sparse term overlap.** BM25's IDF signal needs corpus-level term distribution. Tiny corpora (fewer than a few hundred documents) have unreliable IDF estimates, and short documents with one or two sentences rarely share enough query terms to rank well.

**Cross-lingual retrieval.** BM25 has no concept of semantic equivalence across languages. "Dog" and "chien" have no connection in a BM25 index.

**When recall matters more than precision.** For agent memory systems where missing a relevant document is worse than retrieving an irrelevant one, pure BM25 without query expansion or hybrid fallback underserves the task.

## Hybrid Search: The Standard Deployment Pattern

Production systems almost universally pair BM25 with semantic search. The standard architecture:

1. BM25 retrieves top-K keyword-matched candidates
2. A dense embedding model retrieves top-K semantically similar candidates
3. RRF or a learned reranker merges the lists

RRF formula: `score(d) = Σ 1 / (k + rank(d))` where k=60 is common and rank(d) is the document's position in each individual ranked list. This is robust to score scale differences between BM25 and cosine similarity.

See [Hybrid Search](../concepts/hybrid-search.md) for the full treatment of fusion strategies, reranking, and implementation patterns.

## Key Parameters and Tuning

| Parameter | Default | Effect |
|-----------|---------|--------|
| k₁ | 1.2–2.0 | Higher values reward repeated terms more; lower values saturate faster |
| b | 0.75 | 0 = ignore document length; 1 = full length normalization |
| Field weights (BM25F) | varies | Boost title/heading fields over body text |

Napkin's implementation boosts the `basename` field 2x in MiniSearch, reflecting that a document's filename is more topically concentrated than its body. [Source](../raw/deep/repos/michaelliv-napkin.md) This is standard BM25F practice: weight structural fields higher.

For agent memory corpora, lower b values (0.5–0.6) often work better because shorter notes are not necessarily less relevant — a three-sentence decision record can be more important than a ten-page background document.

## Unresolved Questions

**Optimal chunk granularity.** Napkin's benchmark work shows that per-round notes (~2.5K chars) outperform full session notes (~15K chars) for BM25 retrieval — smaller, focused chunks concentrate term frequency. But the optimal chunk size varies by domain, retrieval pattern, and downstream LLM context budget, and no consensus has emerged on automated chunk-size selection for BM25 corpora.

**BM25 vs. neural sparse retrieval.** Systems like SPLADE (Sparse Lexical and Expansion model) learn sparse representations that expand queries and documents with semantically related terms, addressing BM25's vocabulary gap while preserving the efficiency of sparse retrieval. Whether these hybrid approaches are worth the added complexity for agent memory corpora (typically structured, curated knowledge rather than raw web text) is not settled.

**Score calibration for confidence thresholds.** Napkin hides BM25 scores from agent output deliberately, because numeric scores cause anchoring. But the flip side is that BM25 always returns results — there is no calibrated "I don't know" signal. Napkin reports 50% accuracy on LongMemEval abstention tasks (knowing when not to answer), a direct consequence of this limitation. [Source](../raw/deep/repos/michaelliv-napkin.md) How to derive calibrated confidence from BM25 scores for agent memory systems remains an open engineering problem.

## Alternatives and Selection Guidance

| Use case | Recommendation |
|----------|----------------|
| Exact keyword matching, low infrastructure | BM25 alone |
| General-purpose retrieval with semantic coverage | BM25 + dense embeddings with RRF fusion |
| Novel vocabulary, out-of-distribution terms | BM25 (embeddings fail harder here) |
| Semantic similarity, paraphrase matching | Dense embeddings alone or hybrid |
| Structured knowledge with link graph | BM25 + graph-based reranking (Napkin pattern) |
| Production multi-tenant scale | Elasticsearch/OpenSearch BM25 with optional kNN |

Use [Semantic Search](../concepts/semantic-search.md) when vocabulary gap is the dominant failure mode. Use BM25 when exact term matching matters, infrastructure cost is a constraint, or interpretability of retrieval decisions is required. Use [Hybrid Search](../concepts/hybrid-search.md) when neither alone covers the retrieval distribution.
