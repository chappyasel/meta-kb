# BM25

## What It Is

BM25 (Best Matching 25) is a ranking function from the probabilistic retrieval framework, developed by Robertson and colleagues at City University London in the 1990s. It scores documents against a query by combining term frequency, inverse document frequency, and document length normalization into a single relevance score.

The formula for a query Q with terms q₁...qₙ scoring document D:

```
score(D,Q) = Σ IDF(qᵢ) · [f(qᵢ,D) · (k₁+1)] / [f(qᵢ,D) + k₁·(1 - b + b·|D|/avgdl)]
```

Where:
- `f(qᵢ,D)` is the term frequency of qᵢ in D
- `|D|` is document length, `avgdl` is average document length across the corpus
- `k₁` (typically 1.2–2.0) controls term frequency saturation
- `b` (typically 0.75) controls length normalization
- `IDF(qᵢ) = log((N - n(qᵢ) + 0.5) / (n(qᵢ) + 0.5) + 1)` where N is corpus size and n(qᵢ) is the number of documents containing qᵢ

The "25" refers to the 25th iteration in the Okapi BM series of experiments at City University London. The formula addresses two saturation problems that earlier TF-IDF approaches had: raw term frequency increases score without bound (BM25 saturates it via k₁), and longer documents are unfairly advantaged unless penalized (b controls this).

## Why It Matters in Knowledge Base Systems

BM25 is the standard sparse retrieval baseline for hybrid search systems. Knowledge base retrieval has converged on a two-signal architecture: dense retrieval via embedding cosine similarity for semantic matching, and sparse retrieval via BM25 for lexical matching. The two signals are complementary in a way that matters practically.

Dense retrieval finds semantically similar passages even when vocabulary differs. "Authentication" matches "login" because their embeddings are close. BM25 does not make this connection at all.

BM25 handles exact keyword queries that semantic search muddles. A query for "RFC 2616" or a specific function name like `semaphore_gather` will precisely match documents containing those tokens. Embedding models compress tokens into continuous vectors and can conflate distinct technical terms.

The systems analyzed here all use BM25 as part of hybrid retrieval:

- **Graphiti** combines BM25 fulltext search with cosine similarity and breadth-first graph traversal in its `SearchConfig`. The BM25 component targets "word-level similarity across fact fields, entity names, and community names" — the [Graphiti source](../raw/deep/repos/getzep-graphiti.md) describes this as phi_bm25 running in parallel with phi_cos and phi_bfs.
- **Napkin** uses MiniSearch (a JavaScript BM25 implementation) as its primary retrieval mechanism, with no vector search at all by default. Its benchmark results on LongMemEval — 91% on the S-dataset versus 64% for GPT-4o full context — suggest BM25 on well-structured markdown competes with embedding-based systems for this task type. [Source](../raw/deep/repos/michaelliv-napkin.md)
- **Memento-Skills** implements hybrid retrieval (`multi_recall`) combining `LocalFileRecall` (BM25 over SKILL.md files) and `LocalDbRecall` (semantic vector search via sqlite-vec). [Source](../raw/deep/repos/memento-teams-memento-skills.md)
- **Hipocampus** uses `qmd`, a hybrid BM25 + vector tool, as a fallback search layer behind its primary compaction-tree index. [Source](../raw/deep/repos/kevin-hs-sohn-hipocampus.md)
- **mem-alpha** uses BM25 alongside embedding-based search as dual retrieval modes for its semantic and episodic memory stores. [Source](../raw/deep/repos/wangyu-ustc-mem-alpha.md)

## How It Works in Practice

### Implementations

In Python, `rank_bm25` is the standard library: `BM25Okapi(corpus)` tokenizes documents, builds TF-IDF statistics, and exposes `get_scores(query_tokens)`. JavaScript uses MiniSearch, which supports fuzzy matching and prefix matching on top of BM25 with field-level boosting (Napkin uses `basename` at 2x weight over `content`).

Production systems use database-native BM25 rather than library implementations. Elasticsearch and its Lucene backend have used BM25 as the default similarity function since version 5.0 (2016). Neo4j's fulltext index, used by Graphiti, is backed by Lucene and exposes BM25 scoring. PostgreSQL exposes BM25-style scoring through `ts_rank_cd`. SQLite lacks native BM25 but the `fts5` extension supports it via `bm25()` function.

### Index Construction

BM25 requires an inverted index: a mapping from each unique term to the list of documents containing it, along with per-document term frequencies. For a corpus of N documents with vocabulary V, the index size is O(N × V_avg) where V_avg is average vocabulary per document. For knowledge base systems with thousands of documents, this is modest — under 100MB for most cases.

BM25 indexes rebuild from scratch. There is no incremental update mechanism in the standard formulation: adding a document changes `avgdl` and all IDF values, which requires recalculating scores for every term. Napkin's `search-cache.ts` handles this by storing a fingerprint of all file paths and modification timestamps; any change triggers a full rebuild. For small knowledge bases (thousands of documents), this takes under a second.

### Hybrid Search Integration

The standard pattern for combining BM25 with vector retrieval is Reciprocal Rank Fusion (RRF):

```
RRF(d) = Σ 1 / (k + rankᵢ(d))
```

where k (typically 60) controls rank sensitivity and rankᵢ(d) is document d's position in retrieval list i. RRF requires no parameter tuning, handles score-scale mismatches between BM25 and cosine similarity, and degrades gracefully when one signal is missing. Graphiti uses RRF as one of its reranking strategies. The alternative — learning a linear combination of BM25 and embedding scores — performs better in benchmarks but requires labeled training data.

## Strengths

**Exact match on technical terms.** Function names, error codes, version strings, proper nouns — these retrieve precisely. No embedding model reliably distinguishes `gpt-4.1-mini` from `gpt-4.1-nano` in vector space; BM25 treats them as different strings.

**Zero preprocessing cost.** No embedding model to run, no API calls, no vector dimensionality to choose. Index from text directly. Napkin's benchmark results prove this enables competitive retrieval quality when document structure is good.

**Interpretability.** A BM25 match explains itself: these terms appeared, at these frequencies, in this document. Debugging retrieval failures is tractable.

**No cold start for new terms.** Add a document with new terminology and BM25 retrieves it immediately for exact queries. Embedding-based systems can retrieve it too, but only through semantic proximity to training data.

**Cache-friendly.** Stable document corpora produce stable inverted indexes. Prompt caching systems benefit: a BM25-retrieved passage from a stable knowledge base is the same bytes on repeated queries.

## Limitations

**Vocabulary gap.** BM25's core failure mode: "authn" does not match "authentication," "deploy" does not match "release," "login" does not match "sign in." Synonym expansion (query reformulation, WordNet lookups) partially compensates but adds complexity and latency.

**No semantic understanding.** Two documents with identical term distributions but opposite meanings score identically. BM25 has no concept of negation, context, or implication.

**Query length sensitivity.** Very short queries (one or two words) have high variance. Very long queries dilute the weight of any individual term. Most knowledge base queries fall in the 3–15 word range where BM25 performs well.

**IDF assumes document independence.** The formula treats each document as a bag of words. Near-duplicate documents in the corpus inflate IDF values for shared terms, degrading retrieval quality. Knowledge bases with repeated content (multiple versions of the same document) exhibit this.

**No position information.** BM25 scores term occurrence but not co-occurrence or ordering. "neural network training" and "training neural network" score identically. Phrase queries require additional BM25 variants (BM25F with field structure, or proximity extensions).

## When Not to Use It Alone

BM25 alone fails when:

- The query vocabulary does not overlap with document vocabulary. Cross-lingual retrieval, synonym-heavy domains, and jargon-light user queries all hit this.
- Retrieval requires understanding relationships between concepts, not just matching tokens.
- The corpus is small enough that the IDF signal is weak (under ~50 documents, all terms appear in most documents).

Napkin's LongMemEval benchmark shows a specific case where BM25 alone fails: abstention tasks (knowing when not to answer) score at 50%. BM25 always returns ranked results; it has no calibrated confidence threshold to signal "no relevant document exists." Systems that need to distinguish "no good answer" from "answer available" require either re-ranking with a confidence model or a hybrid approach.

## Relationship to Other Retrieval Approaches

**vs. TF-IDF:** TF-IDF is BM25's predecessor. BM25 adds term frequency saturation (prevents documents from scoring arbitrarily higher by repeating a term) and document length normalization. In practice, BM25 consistently outperforms TF-IDF on IR benchmarks; TF-IDF is largely retained for legacy compatibility.

**vs. Dense Retrieval:** See [Hybrid Retrieval](../concepts/hybrid-retrieval.md) for detailed comparison. The short version: BM25 handles exact match better, dense retrieval handles semantic similarity better. Their errors are largely non-overlapping, which is why combining them improves recall over either alone.

**vs. Sparse Learned Representations (SPLADE, DeepImpact):** These are neural models that produce sparse vector representations in the vocabulary space, improving BM25's vocabulary mismatch problem while retaining the interpretability and index efficiency of sparse retrieval. SPLADE-max achieves higher MRR@10 on MS MARCO than BM25 while remaining sparse. The tradeoff: requires a trained model, has inference cost, and loses BM25's zero-preprocessing advantage.

**vs. BM25F:** BM25F extends the formula to handle multi-field documents (title, body, URL) by weighting term frequencies across fields before applying saturation. Napkin uses MiniSearch's field-level boosting (essentially BM25F with basename at 2x weight) rather than full BM25F.

## Unresolved Practical Questions

**Optimal chunk size for knowledge bases.** Napkin's benchmark design uses per-round notes (~2.5K characters) rather than full session files (~15K characters) and gets better retrieval. The right chunk size is document-type and query-type dependent, and there is no consensus on how to set this without empirical tuning.

**k₁ and b tuning.** Default values (k₁=1.2, b=0.75) work well for English prose. For technical documentation with high term-repetition density (code, API references), lower k₁ may be appropriate. For very short notes, b close to 0 (less length normalization) may help. No standard guidance exists for knowledge base use cases.

**Tokenization for technical content.** Standard tokenizers split on whitespace and punctuation, which breaks technical terms: `gpt-4.1-mini` becomes `gpt`, `4`, `1`, `mini`. Subword tokenization (BPE) handles this better but requires a trained tokenizer. The right tokenizer choice for code and API documentation is underspecified.
