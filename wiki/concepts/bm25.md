# BM25

## What It Is

BM25 (Best Matching 25) is a probabilistic ranking function from the BM family developed by Stephen Robertson and colleagues at City University London, with the key variant published in 1994. It scores a document against a query by combining three signals: how often query terms appear in the document (term frequency), how rare those terms are across the corpus (inverse document frequency), and how the document's length compares to the corpus average (length normalization). The formula:

```
score(D, Q) = Σ IDF(qᵢ) × [f(qᵢ, D) × (k₁ + 1)] / [f(qᵢ, D) + k₁ × (1 - b + b × |D|/avgdl)]
```

Where `f(qᵢ, D)` is the term frequency in document D, `|D|` is document length, `avgdl` is average document length, `k₁` controls term frequency saturation (typically 1.2–2.0), and `b` controls length normalization (typically 0.75). The "25" refers to this being the 25th variant in the BM experimentation series.

BM25 operates entirely on the term level — it requires no learned parameters, no GPU, no embedding model, and no vector index. It tokenizes text, builds an inverted index, and scores at query time using precomputed document frequencies.

## Why It Matters for Agent Knowledge Bases

BM25 is the standard retrieval baseline in [Hybrid Search](../concepts/hybrid-search.md). Every major [RAG](../concepts/rag.md) implementation that claims hybrid retrieval combines BM25 (or a variant) with [vector similarity search](../concepts/vector-database.md). The systems analyzed in this wiki use BM25 at multiple layers:

- **Graphiti** ([project](../projects/graphiti.md)): The search pipeline in `graphiti_core/search/search.py` runs BM25 fulltext search against the graph database alongside cosine similarity and BFS traversal. The three methods target "word similarities, semantic similarities, and contextual similarities" respectively.
- **Memvid**: Ships two BM25 implementations — an embedded Tantivy engine (primary) and a custom fallback in `src/lex.rs` using bincode-serialized token-frequency maps. The Tantivy integration stores index segments inside the `.mv2` binary file, extracting to a temp directory on open.
- **Mem-alpha**: Provides BM25 as one of two memory search modes (alongside embedding cosine similarity), letting the QA model choose the appropriate method per query type.
- **Memento-Skills**: Uses `LocalFileRecall` for BM25 keyword search over SKILL.md files alongside `LocalDbRecall` for vector semantic search in the `skill/retrieval/` module.
- **Hipocampus**: Offers BM25 search as the `qmd` tool fallback, third in the retrieval cascade after ROOT.md triage and manifest-based LLM selection.

## How It Works

### Inverted Index

BM25 requires preprocessing: tokenize all documents, build an inverted index mapping each token to the list of documents containing it (with per-document term frequencies), and compute corpus-level document frequencies. Query time is fast — the scorer only looks up terms present in the query, not all documents.

### Term Frequency Saturation

A key improvement over earlier TF-IDF variants: BM25 applies diminishing returns to term frequency. The `k₁` parameter controls how quickly additional occurrences of a term yield diminishing score boosts. With `k₁ = 0`, term frequency is ignored entirely. As `k₁ → ∞`, BM25 approaches linear TF scoring. The standard value `k₁ = 1.5` means a term appearing 10 times in a document scores only about 4× better than one appearing once, rather than 10×.

### Length Normalization

The `b` parameter penalizes longer documents. Without normalization, longer documents accumulate more term matches simply from having more words. With `b = 0.75` (standard), a document twice the average length gets a modest penalty that prevents length bias while preserving the signal that longer documents with the same term frequency are providing richer context.

### IDF Weighting

BM25 uses a smoothed IDF formula: `IDF(q) = log((N - n(q) + 0.5) / (n(q) + 0.5) + 1)` where N is the total document count and `n(q)` is the number of documents containing term q. Rare terms score higher. Common terms (appearing in most documents) approach zero IDF weight.

## Strengths

**Exact keyword retrieval**: BM25 finds documents containing the query terms, even when embedding models would fail. Product codes, identifiers, technical terms, names, and rare vocabulary all retrieve reliably.

**Zero infrastructure**: No GPU, no model download, no embedding API. BM25 runs on any CPU with modest memory. [Elasticsearch](../projects/elasticsearch.md) and [SQLite](../projects/sqlite.md) (via FTS5) both ship BM25 implementations.

**Interpretable scoring**: You can inspect why a document scored as it did — which terms matched, their frequencies, and the IDF weights. This is not possible with [embedding models](../concepts/embedding-model.md).

**No training, no drift**: BM25 parameters are set once and remain stable. Embeddings can drift when you switch models or fine-tune, requiring re-indexing. BM25 only requires re-indexing when the corpus changes.

**Fast updates**: Adding a document to a BM25 index requires updating term frequencies for the document's tokens. Adding a document to an HNSW vector index requires graph reconnection, which is more expensive at scale.

## Critical Limitations

**Vocabulary mismatch failure**: BM25 cannot match synonyms, paraphrases, or cross-lingual queries. If a document says "automobile" and the query says "car," BM25 scores zero for that term. This is the retrieval gap that [embedding models](../concepts/embedding-model.md) fill.

**Concrete failure mode**: An agent asking "what does Alice do for work?" will score poorly against a document containing "Alice is employed as a software engineer at Acme" if the agent's query tokenizes to `what,alice,work` — none of which appear literally in the relevant sentence fragment. Graphiti's paper quantifies this: their hybrid search (BM25 + cosine + BFS) substantially outperforms any single method, with the temporal reasoning improvement (+38.4%) coming specifically from the non-BM25 components.

**Unspoken infrastructure assumption**: BM25 assumes a stable, tokenizable text corpus. It works poorly on structured data (JSON, tables, code with identifiers), heavily abbreviated text, and multilingual content where token boundaries differ by language. Memvid's SKILL.md files note that the keyword-dense format in compaction nodes is specifically designed to maximize BM25 recall — information-dense prose naturally loses terms that BM25 would need.

**No semantic ranking**: Two documents, one highly relevant and one topically adjacent, receive the same score if they contain the same query terms at the same frequency. Ranking quality plateaus on semantic tasks where meaning matters more than vocabulary overlap.

## When NOT to Use BM25 Alone

**Conversational memory with paraphrase-heavy queries**: When users express concepts in varied ways across sessions, BM25 misses connections that [embedding similarity](../concepts/embedding-model.md) finds. Graphiti's DMR benchmark (94.8% accuracy) depends on the hybrid pipeline, not BM25 alone.

**Cross-lingual knowledge bases**: BM25 indexes tokens. A knowledge base in English with queries in Chinese or Spanish will fail to retrieve unless translation is part of the pipeline. Hipocampus handles this in ROOT.md's Topics Index by manually including cross-language keyword variants, but BM25 itself provides no cross-lingual capability.

**Graph traversal tasks**: BM25 cannot follow relationships. "Who works with Alice's manager?" requires traversing graph edges, which BM25 has no concept of. This is why Graphiti adds BFS graph traversal as a third search signal alongside BM25 and cosine similarity.

**Highly paraphrased corpora**: If your knowledge base uses consistent technical terminology, BM25 performs well. If the same concept appears under many surface forms (as in natural language conversations), BM25 recall drops and hybrid search becomes necessary.

## Integration Patterns

### Standard Hybrid Pipeline

The dominant pattern across systems is BM25 + vector search + [Reciprocal Rank Fusion (RRF)](../concepts/reciprocal-rank-fusion.md) for score fusion:

1. Run BM25 query against inverted index → ranked list A
2. Run embedding similarity query against vector index → ranked list B
3. Merge A and B using RRF: `score(d) = Σ 1 / (k + rank(d))` with k=60

Graphiti's `COMBINED_HYBRID_SEARCH_CROSS_ENCODER` config adds a fourth step: neural cross-encoder reranking of the merged candidate set.

### Implementation Backends

- **Tantivy** (Rust): Used by Memvid. Full-featured BM25 with field weighting, stored in embedded segments. Requires temp directory during session.
- **Elasticsearch/OpenSearch**: Distributed BM25 with tunable field weights, sharding, and real-time indexing. Standard for production multi-tenant deployments.
- **SQLite FTS5**: Ships in SQLite's standard distribution. Simple, portable, adequate for smaller corpora. Used in [Zep](../projects/zep.md).
- **rank_bm25** (Python): Pure Python implementation used by Mem-alpha for in-memory BM25 over semantic and episodic memory entries.
- **Lucene-based** ([Elasticsearch](../projects/elasticsearch.md), Solr): Java implementations with extensive tuning options. Standard in enterprise search.

### Tokenization Matters

BM25 results depend on the tokenizer. Lowercasing, stemming, stopword removal, and n-gram tokenization all affect which terms enter the inverted index. A query for "running" won't match a document containing "run" unless the tokenizer stems both to a common root. Systems like Graphiti rely on the graph database's native fulltext indexer (Neo4j uses Lucene, FalkorDB uses its own FTS implementation), which means BM25 behavior varies by backend.

## Relationship to Other Concepts

BM25 pairs with [Hybrid Search](../concepts/hybrid-search.md) as the sparse retrieval component. It functions as the complement to [Embedding Model](../concepts/embedding-model.md)-based dense retrieval. [Reciprocal Rank Fusion](../concepts/reciprocal-rank-fusion.md) is the standard method for merging BM25 and vector search results. In [Agentic RAG](../concepts/agentic-rag.md) systems, BM25 typically handles the first-pass retrieval before reranking.

In [Context Management](../concepts/context-management.md), BM25 enables [Progressive Disclosure](../concepts/progressive-disclosure.md) by retrieving the most relevant documents before deciding what to load into the [Context Window](../concepts/context-window.md).

## Unresolved Questions

**Optimal k₁ and b for agent memory corpora**: The standard values (k₁=1.5, b=0.75) were tuned on news/web corpora. Agent memory logs (short conversational snippets, repeated entity names, heavy entity overlap across documents) may need different parameter settings. No published benchmarks compare BM25 hyperparameter sensitivity specifically on agent memory datasets.

**Field-weighted BM25 for structured knowledge**: Systems like Graphiti search over both fact descriptions and entity names simultaneously. Whether to apply uniform BM25 or field-specific weights (e.g., entity name matches weighted higher than fact description matches) is an open tuning question with no standard answer.

**Sparse vs. learned sparse retrieval**: SPLADE and similar learned sparse methods produce BM25-compatible inverted indices but with learned term weights. Whether these outperform classical BM25 on conversational memory retrieval at the costs and latencies relevant to agent systems remains unquantified in the published literature.

## Alternatives

- **[Embedding Model](../concepts/embedding-model.md) + vector search**: Use when queries and documents use varied vocabulary, paraphrasing is common, or cross-lingual retrieval is needed. Higher infrastructure cost, no exact keyword matching.
- **[Hybrid Search](../concepts/hybrid-search.md)**: Use BM25 + vector search together for most production RAG systems. The marginal cost of adding BM25 to an existing vector pipeline is low, and recall improvements are consistent.
- **[Vector Database](../concepts/vector-database.md) alone**: Acceptable for purely semantic retrieval over homogeneous corpora. Loses the exact-match guarantees that BM25 provides.
