# LLM-as-a-Judge

## What It Is

LLM-as-a-Judge is an evaluation approach where one language model scores, critiques, or compares outputs generated by another model or agent. Instead of requiring human reviewers for every output, or relying on exact-match metrics that require labeled ground truth, you route outputs through a judge model that applies a rubric and returns a score, a binary pass/fail, or natural language feedback.

The approach shows up across three distinct contexts: offline evaluation (testing prompt or model changes before shipping), online monitoring (scoring production traffic continuously), and as a reward signal inside autonomous improvement loops (where agent outputs are scored, the failures trigger prompt refinement, and the cycle repeats).

## Why It Matters

Traditional NLP metrics like BLEU or ROUGE measure surface similarity to a reference string. They fail on tasks where multiple valid outputs exist and where quality depends on semantic content, instruction following, or domain correctness. Human review scales poorly and introduces annotator inconsistency. LLM-as-a-Judge fills the gap: it can assess nuanced criteria like regulatory compliance, factual faithfulness, completeness, and tone at arbitrary scale.

The GraphRAG paper used LLM-as-a-Judge head-to-head comparisons (5 runs averaged) to evaluate comprehensiveness, diversity, empowerment, and directness across 125 sensemaking questions per dataset, producing the 72-83% win rates reported for GraphRAG vs. naive RAG — though the paper itself acknowledges this method may systematically favor longer, more structured responses. [Source](../raw/deep/papers/edge-from-local-to-global-a-graph-rag-approach-to-quer.md)

## How It Works

### Anatomy of a Judge Call

A judge call requires three things: the original input (so the judge has context), the output being evaluated, and a rubric specifying what "good" looks like. The rubric can be a numeric scale with anchored descriptions, a binary pass/fail with criteria, or a comparative "which is better" prompt.

The OpenAI self-evolving agents cookbook implements this as a `score_model` grader with an explicit 0–1 scale and anchored descriptions at each level:

```
"A score of 1 means the summary is almost flawless: comprehensive, 
highly faithful, technically accurate..."
"0.0-0.3 means there are major omissions, misunderstandings, or a 
failure to capture the most important technical content."
```

[Source](../raw/articles/developers-openai-com-self-evolving-agents-a-cookbook-for-autonomous-a.md)

The judge model receives `item.section` (the source text) and `sample.output_text` (the generated summary) and returns a single float. The pass threshold is 0.85.

### Deployment Patterns

**Offline evaluation**: Run the judge against a curated dataset before shipping a prompt or model change. LangSmith's annotation queues let teams build offline datasets from production traces with human labels, then run judge-scored evals against those labeled examples to produce before/after comparisons. [Source](../raw/articles/langchain-com-the-agent-improvement-loop-starts-with-a-trace.md)

**Online monitoring**: Run judges continuously on sampled production traffic. LangSmith's online evaluators support this pattern — configuring them to score all traces, a random sample, or a filtered subset based on criteria like low user ratings. This catches quality drift and surfaces failure patterns without blocking the request path.

**Reward signal in retraining loops**: The langgraph-reflection architecture uses an LLM judge as the exit condition for an iterative critique loop. The main agent generates output; the judge evaluates it; if the score fails, the judge's critique becomes a new user message that the main agent must address. The loop terminates when the judge finds nothing to critique or returns no messages. [Source](../raw/repos/langchain-ai-langgraph-reflection.md)

The self-evolving agents cookbook combines the judge with three other graders (chemical name presence, word length deviation, cosine similarity) inside a feedback loop. When all graders pass 75% of the time and the average score exceeds 0.85, the loop accepts the current prompt. When graders fail, a metaprompt agent reads the judge's reasoning and generates an improved prompt. [Source](../raw/articles/developers-openai-com-self-evolving-agents-a-cookbook-for-autonomous-a.md)

### Judge Calibration

A judge without calibration drifts from human judgment. The standard calibration approach: route a sample of traces through both the automated judge and human reviewers, identify disagreements, and use those labeled disagreements to tune the judge prompt. LangSmith formalizes this with annotation queues specifically for "aligning online evaluators" — reviewers label traces where the automated score looks wrong, and those examples become calibration data. [Source](../raw/articles/langchain-com-the-agent-improvement-loop-starts-with-a-trace.md)

GEPA (Genetic-Pareto prompt optimization) uses a similar pattern: it runs the judge on training examples, generates reflective feedback, proposes prompt revisions, and validates on a held-out set — separating training and validation signals to reduce overfitting to the judge's preferences. [Source](../raw/articles/developers-openai-com-self-evolving-agents-a-cookbook-for-autonomous-a.md)

## Implementation Details

### Score Model vs. Binary vs. Comparative

Three formats appear in practice:

**Numeric scoring** (`score_model` in OpenAI Evals): Returns a float in `[0, 1]`. Useful when you need a gradient for optimization. Requires anchored rubrics at each scale point or scores cluster at 0.5 and 1.0.

**Binary pass/fail**: Returns True/False against a threshold. Simpler to reason about and calibrate. The langgraph-reflection `create_llm_as_judge` from `openevals` uses `feedback_key="pass"` — the judge's score on that key determines whether the critique loop continues or terminates.

**Comparative (A vs. B)**: Presents two outputs and asks the judge which is better, optionally with rationale. Used in the GraphRAG paper for head-to-head evaluation. Reduces absolute scale calibration problems but requires paired outputs and doesn't produce individual scores.

### Rubric Design

The rubric is where most judge failures originate. Common problems:

- Rubrics that reward length. Judges trained on RLHF data often associate longer responses with higher quality. The GraphRAG paper explicitly flags this: LLM-as-judge may systematically prefer longer responses, which inflates GraphRAG's comprehensiveness scores.
- Rubrics without negative examples. Judges given only positive descriptions of quality apply them loosely.
- Under-specified criteria. "Evaluate whether the summary is accurate" is not a rubric. "A score below 0.5 means technical facts from the source are misstated or absent from the summary" is.

The self-evolving agents cookbook addresses this with multi-grader ensembling: the LLM judge handles nuanced quality signals while deterministic graders (exact chemical name matching, word count deviation) catch failures the judge would likely miss or score incorrectly.

### Cost and Latency

Running a judge model adds one LLM call per evaluation. For online monitoring on high-volume systems, this cost compounds. Mitigations:

- Sample rather than score every trace (random or stratified by confidence)
- Use a smaller judge model for obvious failures, routing edge cases to a larger model
- Cache judge results for identical inputs
- Reserve LLM judging for qualitative dimensions; use deterministic checks for binary criteria

## Strengths

**Flexible rubric application**: A judge can evaluate instruction following, domain correctness, tone, and completeness in a single call — criteria that would require separate rule-based systems or labeled datasets to evaluate otherwise.

**Actionable feedback**: Unlike numeric metrics, a judge can return natural language explanations of failures. The self-evolving loop uses this directly: the judge's reasoning feeds into the metaprompt agent as the basis for prompt revision. [Source](../raw/articles/developers-openai-com-self-evolving-agents-a-cookbook-for-autonomous-a.md)

**No labeled ground truth required for criteria-based evaluation**: When multiple valid outputs exist, you can evaluate against criteria rather than a reference string. This makes LLM-as-a-Judge applicable to generative tasks where exact-match metrics fail entirely.

## Critical Limitations

**Position and length bias**: Judge models exhibit systematic preferences for longer responses and for whichever option appears first in comparative evaluations. The GraphRAG authors flag this explicitly but do not correct for it. Any benchmark using LLM-as-a-Judge without position-debiasing (presenting each pair in both orders and averaging) should be read with skepticism.

**Compounding hallucination in automated loops**: In the self-evolving agents pattern, the judge scores summaries of LLM-extracted entities — meaning errors compound across two LLM calls. The judge may score a plausible-sounding but incorrect chemical name as passing if the rubric doesn't include exact-match verification. The multi-grader approach (chemical name exact match + judge) addresses this, but the failure mode exists in any system where the judge operates on already-processed content.

**Infrastructure assumption**: LLM-as-a-Judge in production assumes a reliable, low-latency path to the judge model. For online scoring of production traffic, this means the evaluation pipeline must handle the same availability and latency SLOs as the main application. If the judge call fails or times out, you lose observability silently unless you instrument for it.

## When NOT to Use It

**When deterministic checks are sufficient**: Schema validation, regex-matchable formats, exact-match conditions, and business rule compliance are cheaper and more reliable as code-based checks. Routing these through an LLM judge adds cost and introduces false negatives.

**For high-stakes domain correctness without domain-expert calibration**: A legal or medical agent requires reviewers who understand what "correct" means in context. An LLM judge applied to these domains without calibration against domain-expert labels will miss the failure modes that matter most. The LangChain article puts it directly: "A legal research agent that cites plausible-sounding but inaccurate precedents might fool an LLM judge." [Source](../raw/articles/langchain-com-the-agent-improvement-loop-starts-with-a-trace.md)

**When the judge and generator share the same training distribution**: A judge model fine-tuned on the same data as the model it evaluates will systematically miss the failure modes that data doesn't cover. This is a particular concern when using the same model family for both generation and evaluation.

**When you need reproducible benchmarks for external publication**: Self-reported LLM-as-a-Judge numbers without human validation are not independently verifiable. The GraphRAG 72-83% comprehensiveness win rates are self-reported and use the same evaluation methodology across compared systems, which partially controls for judge bias but doesn't eliminate it.

## Unresolved Questions

**Optimal judge model size**: There is no consensus on whether the judge model needs to be larger than, equal to, or can be smaller than the model being judged. The self-evolving cookbook uses `gpt-4.1` as judge for outputs from `gpt-5`, but the tradeoffs of using smaller models (cost) vs. larger models (quality) at scale are domain-dependent and rarely measured.

**Judge drift over time**: If the judge model is updated (new version, new fine-tune), previously calibrated thresholds may no longer apply. None of the sources address how to handle judge model versioning in production evaluation pipelines.

**Feedback loop stability**: In autonomous improvement loops, a judge that produces slightly wrong scores can drive the optimizer in the wrong direction. The self-evolving cookbook addresses this with multi-grader ensembling and a held-out validation set, but the conditions under which LLM-judge-driven optimization diverges vs. converges are not formally characterized.

## Alternatives

**Deterministic graders**: Use when criteria are binary and expressible as code. Faster, cheaper, and fully reproducible. Best combined with LLM-as-a-Judge rather than replaced by it.

**Human annotation queues**: Use when domain correctness requires expertise the judge doesn't have, or when you need calibration data. Human annotation is the ground truth that calibrates the judge, not a replacement for it.

**Reference-based metrics (BLEU, ROUGE, BERTScore)**: Use when labeled reference outputs exist and surface similarity to that reference is a valid proxy for quality. Unsuitable for open-ended generation.

**Constitutional AI / self-critique without a separate model**: Use when the goal is single-model output refinement and you don't have a separate judge budget. Less reliable than using a separate model but zero additional infrastructure cost.
