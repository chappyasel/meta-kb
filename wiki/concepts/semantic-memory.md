# Semantic Memory

## What It Is

Semantic memory is the component of an agent's memory system that stores general facts, concepts, and world knowledge — information that persists independently of when or how it was learned. The term comes from cognitive psychology, where semantic memory (knowing that Paris is in France) is distinguished from episodic memory (remembering the day you learned that) and procedural memory (knowing how to read a map).

In AI agent systems, semantic memory solves a concrete engineering problem: LLMs have finite context windows, conversations end, and agents restart, but accumulated knowledge about the world should persist across all of these. Semantic memory is the layer where that knowledge lives.

The distinction matters operationally. An agent might learn that a user works in oncology research, that a particular API returns rate-limit errors above 100 requests per second, or that a codebase uses snake_case for internal variables. None of these are episodic events — they are stable facts that should inform every future interaction. Storing them in semantic memory means the agent retrieves them on demand rather than rediscovering them from conversation history each session.

## How It Works

### Storage Representations

Semantic memory takes three main forms in practice:

**Flat fact strings.** The simplest implementation stores semantic knowledge as a list of text statements: "User is a cardiologist at UCSF," "Project uses Python 3.11," "API rate limit is 100 req/sec." [Mem0](../projects/mem0.md) uses this approach. Facts are embedded and stored in a vector database. Retrieval runs a similarity search against the query, returning the top-k matching facts. The approach is easy to implement and works well for simple personal assistant use cases but loses relational structure — facts about Alice, Bob, and their relationship to Acme Corp all float independently.

**Structured triples.** More sophisticated systems represent semantic knowledge as entity-relation-entity triples: (Alice, WORKS_AT, Acme Corp), (Acme Corp, OPERATES_IN, pharmaceutical sector). [Graphiti](../projects/graphiti.md) implements this with typed `EntityNode` and `EntityEdge` objects. Each edge carries a natural-language fact description alongside its structured form, plus temporal validity bounds (`valid_at`, `invalid_at`). This representation supports queries that flat strings cannot answer: "What entities connect Alice to the pharmaceutical sector within two hops?"

**Community summaries.** Graphiti also builds a third tier above individual entities: `CommunityNode` objects generated by label propagation clustering across the entity graph, with LLM-generated summaries. These summaries compress large clusters of related entities into searchable paragraphs. A query about oncology research might retrieve the oncology community summary before drilling into individual facts — similar to how a human might recall a domain of knowledge before specific details.

### How Facts Reach Semantic Memory

Most systems use LLM-based extraction to decide what qualifies as semantic knowledge.

Graphiti's pipeline for a new conversation turn runs five stages: extract candidate entities, deduplicate against existing entities via hybrid search plus LLM adjudication, extract fact triples, resolve contradictions with existing edges, then update entity summaries. The extraction prompt explicitly instructs the LLM on what not to extract — pronouns, abstract nouns, bare kinship terms. This specificity reduces noise but can miss valid facts in domain-specific text. [Source](../raw/deep/repos/getzep-graphiti.md)

[Mem0](../projects/mem0.md) takes a simpler path: a single LLM call decides which of the new memories supersede, contradict, or add to existing ones, returning integer IDs of affected facts. This is faster per operation but produces less structured output.

[Mem-alpha](../raw/deep/repos/wangyu-ustc-mem-alpha.md) trains a 4B-parameter model with reinforcement learning to make these extraction decisions autonomously. The reward signal is QA accuracy on questions answered using only the constructed memory — the agent learns which facts to retain by seeing whether retaining them actually helps answer questions. The trained model generalizes from 30K-token training contexts to 400K+ token sequences, suggesting that memory construction is a general skill rather than a length-specific heuristic.

[OpenMemory](../raw/deep/repos/caviraoss-openmemory.md) uses regex-based routing to classify memories into sectors (episodic, semantic, procedural, emotional, reflective) without any LLM call. "How to" patterns route to procedural; factual definitions route to semantic. This is fast and costs nothing but fails on content that does not match its English-centric patterns.

### Retrieval

The dominant retrieval pattern is [Hybrid Retrieval](../concepts/hybrid-retrieval.md): combining dense vector similarity (embedding search) with sparse keyword search ([BM25](../concepts/bm25.md)), then optionally reranking with a cross-encoder or [Reciprocal Rank Fusion](../concepts/hybrid-retrieval.md).

Graphiti adds graph traversal as a third retrieval signal: breadth-first search from recently mentioned entities surfaces contextually related facts that neither embedding similarity nor keyword overlap would find. A question about Alice's current project might not lexically match facts about her employer, but two hops through the entity graph would connect them.

Mem-alpha provides both BM25 and embedding search as tools the agent can invoke dynamically during QA. The agent decides which to use based on the query — keyword search for proper nouns, embedding search for paraphrase-heavy queries.

### Temporal Handling

Semantic facts change over time. Alice leaves Acme and joins BioTech. The API rate limit increases. The correct Python version for a project changes.

Systems differ significantly in how they handle this:

**Edge invalidation with timestamps.** Graphiti marks contradicted facts as expired (`expired_at`) rather than deleting them. Each edge carries both event-time validity (`valid_at`, `invalid_at` — when the fact was true in the world) and transaction-time validity (`t'_created`, `t'_expired` — when the system knew it). This bi-temporal model allows answering "What did we believe about Alice's employer in January?" independently from "Where does Alice work now?" [Source](../raw/deep/papers/rasmussen-zep-a-temporal-knowledge-graph-architecture-for-a.md)

**Overwrite with history.** Mem0 resolves contradictions by deciding which memory supersedes which, updating the stored fact. Prior states are not preserved.

**Confidence decay.** OpenMemory applies exponential decay to facts over time, with sector-specific rates — semantic facts decay at lambda=0.005 (slowest) while emotional memories decay at lambda=0.02. Facts that are frequently accessed or recently seen decay more slowly. This is biologically inspired but the parameter values are hand-tuned rather than empirically validated.

The bi-temporal approach is most powerful for domains where historical states matter. For simple personal assistant use cases, it adds infrastructure complexity without proportional benefit.

## Who Implements It

**[Letta](../projects/letta.md)** (formerly MemGPT) gives the agent direct read/write access to a core memory block (always in context) and archival memory (vector search on demand). Archival memory functions as semantic memory: facts persist across sessions and the agent retrieves them when the core block lacks relevant context.

**[Mem0](../projects/mem0.md)** manages semantic memory automatically, running extraction and deduplication on each conversation turn. The developer does not write memory operations; Mem0 decides what to store and update.

**[Graphiti](../projects/graphiti.md) / [Zep](../projects/zep.md)** implement the most complete semantic memory architecture, with typed entities, structured fact triples, bi-temporal edges, and community summarization. Zep exposes this as a managed service; Graphiti is the open-source library underneath.

**[LangChain](../projects/langchain.md) and [LangGraph](../projects/langgraph.md)** provide memory stores that can function as semantic memory backends, with pluggable vector database backends ([ChromaDB](../projects/chromadb.md), [Qdrant](../projects/qdrant.md), [Pinecone](../projects/pinecone.md)) for retrieval.

**Mem-alpha** demonstrates that a small RL-trained model can outperform rule-based systems at deciding what to store in semantic memory and when — though it requires training infrastructure (4+ GPUs, the verl framework) rather than being a drop-in library. [Source](../raw/deep/repos/wangyu-ustc-mem-alpha.md)

## Relationship to Other Memory Types

Semantic memory is one part of the broader [Agent Memory](../concepts/agent-memory.md) taxonomy:

- **[Episodic Memory](../concepts/episodic-memory.md)** stores specific events — what happened in session 3 on Tuesday. Semantic memory generalizes from episodes: many observations that "Alice usually asks for concise responses" produce the semantic fact "Alice prefers concise responses."
- **[Core Memory](../concepts/core-memory.md)** (in Letta's architecture) is a small always-present scratchpad in the system prompt. Semantic memory is the larger, searchable store that core memory draws from.
- **[Procedural Memory](../concepts/procedural-memory.md)** stores how to do things — agent workflows, tool usage patterns. Semantic memory stores what is true; procedural memory stores how to act.
- **[Retrieval-Augmented Generation](../concepts/rag.md)** retrieves from a static document corpus. Semantic memory retrieves from a dynamic, agent-updated knowledge store. The difference is who writes it: RAG reads documents humans created, semantic memory accumulates facts the agent learned through interaction.

## Failure Modes

**Extraction errors compound.** Every fact in semantic memory arrived through an extraction pipeline with nonzero error rate. Incorrect facts sit alongside correct ones with no inherent signal distinguishing them. Over time, incorrect facts accumulate and corrupt downstream reasoning. Unlike [Episodic Memory](../concepts/episodic-memory.md), where the original conversation is still available for verification, semantic memory has abstracted away its source.

**Schema mismatch across sessions.** If extraction prompts or entity type definitions change between deployment versions, facts extracted under the old schema coexist with facts from the new schema. A fact about "Alice (Person)" and another about "Alice Smith (Employee)" may refer to the same person but appear as separate nodes, fragmenting the graph.

**Cold start degradation.** A new agent starts with empty semantic memory. For the first several sessions, the agent lacks context that a human expert would have internalized. This is most acute for domain-specific use cases where the agent needs accumulated knowledge to be useful.

**Retrieval failure under vocabulary mismatch.** If a user asks "Where does Alice work?" but the stored fact says "Alice is employed at Acme Corp," embedding similarity usually bridges this. But if the fact is stored as "Acme Corp employs Alice," directionality may break retrieval. BM25 will miss "Alice" in that phrasing. Graph traversal from Alice's entity node would find it, which is part of why graph-based retrieval outperforms pure vector search for semantic memory.

**Stale facts without invalidation.** In systems without temporal invalidation, superseded facts remain valid. An agent that stored "Alice's phone is 555-1234" in 2023 will confidently provide that number in 2025 even if Alice changed phones and the agent was told about it — if the system overwrote rather than appended, and the overwrite failed for any reason.

Graphiti's -17.7% regression on single-session-assistant tasks (questions about what the assistant itself said) reveals a specific gap: semantic memory systems optimized for extracting user-stated facts tend to under-index assistant-generated content. In agentic settings where the agent's prior reasoning and outputs are important context, this is a material limitation. [Source](../raw/deep/papers/rasmussen-zep-a-temporal-knowledge-graph-architecture-for-a.md)

## When to Use Semantic Memory

Semantic memory makes sense when:

- Agent sessions are long-running or span multiple conversations, and facts learned early should influence behavior later
- The domain has stable world knowledge that does not exist in training data (organization-specific facts, project-specific conventions, user preferences)
- Retrieval latency matters: 3 seconds querying semantic memory beats 31 seconds processing a full 115K-token conversation history

Skip it when:

- Conversations are short and self-contained — the full conversation fits in context without degrading response quality
- Extracted facts would require high-quality LLM adjudication at a per-message cost that exceeds budget (each Graphiti ingestion requires 4-5 LLM calls)
- The domain has rapidly changing facts where stale-fact risk outweighs retrieval benefit
- You need to query the corpus with complex structured filters — a traditional database beats semantic memory for anything beyond fuzzy semantic search

## Performance Evidence

The strongest benchmark evidence for semantic memory comes from Zep/Graphiti on LongMemEval (115K-token conversations):

- gpt-4o accuracy: 71.2% with Zep vs 60.2% without, an 18.5% gain
- gpt-4o-mini accuracy: 63.8% with Zep vs 55.4% without, a 15.2% gain
- Latency: ~3.2 seconds with Zep vs ~31 seconds processing the full conversation (90% reduction)
- Temporal reasoning questions: +48.2% improvement (validates bi-temporal indexing)

These are self-reported by the Zep team (arXiv:2501.13956). No independent replication exists yet. [Source](../raw/deep/papers/rasmussen-zep-a-temporal-knowledge-graph-architecture-for-a.md)

Mem-alpha reports state-of-the-art results on MemoryAgentBench over Mem0, Letta, and long-context baselines, with training-to-inference generalization from 30K to 400K+ tokens. These are self-reported from an academic preprint (arXiv:2509.25911). [Source](../raw/deep/repos/wangyu-ustc-mem-alpha.md)

## Unresolved Questions

**Optimal extraction granularity.** No published work establishes whether storing more specific facts (Alice works at Acme Corp, oncology division, New York office) outperforms coarser facts (Alice works at Acme) across retrieval tasks. More specific facts support precise queries but generate more retrieval candidates to rerank.

**Conflict resolution under authority.** Most systems resolve contradictions by recency — newer facts supersede older ones. This fails for domains where older authoritative sources (medical records, legal documents) should take precedence over more recent conversational references.

**Memory for agent outputs.** Systems consistently perform worse at indexing and retrieving the agent's own prior statements compared to user-provided facts. No published system has solved this cleanly.

**Long-term graph maintenance.** Graph-based semantic memory grows monotonically. Graphiti keeps expired edges rather than deleting them. No published work establishes when graph cleanup is needed, how to detect community-structure drift, or what compaction costs look like at millions of entities.
