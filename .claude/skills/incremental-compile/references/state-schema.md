# Incremental State Schema

File: `build/incremental-state.json`

Generated by the script pipeline (`bun run compile`) after every compilation.
Read by the `incremental-compile` skill to detect what changed.

## Schema

```typescript
interface IncrementalState {
  version: 1;
  config_hash: string;
  source_hashes: Record<string, string>;
  entity_input_hashes: Record<string, string>;
  previous_entity_ids: string[];
  synthesis_hashes: Record<string, string>;
  last_compiled: string;
}
```

## Field Descriptions

### `config_hash`
SHA-256 of `config/domain.ts` file contents + compilation thresholds
(`FULL_ARTICLE_THRESHOLD_REFS=3`, `FULL_ARTICLE_THRESHOLD_RELEVANCE=7.0`) +
model version strings (`claude-haiku-4-5`, `claude-sonnet-4-6`).

If this hash differs from the current config, the entire wiki must be
recompiled — incremental mode cannot safely skip passes when the domain
definition or thresholds changed.

### `source_hashes`
Maps source paths (relative to `raw/`, e.g. `repos/mem0ai-mem0.md`) to
SHA-256 of their file contents at last compilation.

Used to detect added, modified, and deleted sources. A source is:
- **Added** if its path exists on disk but not in this map
- **Modified** if its path exists in both but hashes differ
- **Deleted** if its path exists in this map but not on disk

### `entity_input_hashes`
Maps entity IDs (e.g. `mem0`, `graphiti`) to a hash of the inputs that
reflect changed evidence:

```
SHA-256(bucket + "|" + source_refs.sort().join(","))
```

Only `bucket` and `source_refs` are hashed. Description, aliases, and
neighborIds are excluded because they change cosmetically across entity
re-resolution without reflecting new evidence, causing cascading
false-dirty detection (e.g., 131/131 entities dirty when only 7 sources
changed).

An entity is dirty (needs card regeneration) if its input hash changed or
if it doesn't appear in this map (new entity).

### `previous_entity_ids`
List of all entity IDs from the last compilation. Serves two purposes:
1. **Deleted entity detection**: Entities in this list but not in the current
   compilation have been removed — their wiki pages should be deleted.
2. **Entity ID anchoring** (Pass 1b): These IDs are sent to Sonnet during
   entity resolution as "existing entities." When a raw mention matches a
   previous entity, Sonnet reuses the exact ID instead of generating a new
   slug. This prevents entity ID churn across compilations (e.g., `mem0`
   stays `mem0` instead of becoming `mem0ai` in a later run).

### `synthesis_hashes`
Maps bucket IDs (e.g. `agent-memory`) to SHA-256 of the synthesis article
content at last compilation.

Used to decide whether field-map.md needs regeneration — if no synthesis
hashes changed, the field map is still current.

### `last_compiled`
ISO 8601 timestamp of when the state was saved.

## Example

```json
{
  "version": 1,
  "config_hash": "a1b2c3d4...",
  "source_hashes": {
    "repos/mem0ai-mem0.md": "e5f6a7b8...",
    "tweets/karpathy-llm-knowledge.md": "c9d0e1f2...",
    "articles/gustycube-markdown-memory.md": "3a4b5c6d..."
  },
  "entity_input_hashes": {
    "mem0": "7e8f9a0b...",
    "graphiti": "1c2d3e4f..."
  },
  "previous_entity_ids": ["mem0", "graphiti", "letta", "napkin"],
  "synthesis_hashes": {
    "knowledge-substrate": "9e0f1a2b...",
    "agent-memory": "5a6b7c8d...",
    "context-engineering": "c3d4e5f6...",
    "agent-architecture": "a7b8c9d0...",
    "multi-agent-systems": "e1f2a3b4...",
    "self-improving": "5c6d7e8f..."
  },
  "last_compiled": "2026-04-06T20:00:00.000Z"
}
```
