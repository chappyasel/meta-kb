## Architecture Overview

Napkin is a local-first, file-based knowledge system designed specifically as agent memory. It operates directly on markdown files in an Obsidian-compatible vault structure, requiring no servers, databases, or external services. The architecture is a TypeScript CLI + SDK that can be used both as a command-line tool (`napkin-ai`) and as a programmatic library (`import { Napkin } from "napkin-ai"`).

Created by Michael Livshits, napkin represents an opinionated bet against the conventional wisdom that embedding-based RAG is necessary for knowledge retrieval. The core argument: you have a model that understands semantics, and you bolt on a smaller, dumber model (an embedding model) to pre-filter information before the smart one ever sees it. The embedding model makes the retrieval decisions, which is backwards. Instead, give the big model a map of the knowledge base and let it navigate using its full reasoning capability, not cosine similarity from a smaller model.

The codebase follows a strict three-layer separation:

1. **Core modules** (`src/core/`) -- Pure logic that returns typed data. Never calls `console.log`, `process.exit`, or imports output utilities. Functions like `searchVault()`, `getOverview()`, `readFile()` return structured objects.
2. **Command wrappers** (`src/commands/`) -- Thin CLI layer that parses arguments, calls SDK methods, and formats output for terminal/JSON/quiet modes.
3. **SDK class** (`src/sdk.ts`) -- The `Napkin` class wraps core modules into a clean API surface. Always works, even creating a bare vault if needed.

The vault structure places `.napkin/` as config-only (holding `config.json` that syncs bidirectionally with `.obsidian/`), while content lives in the project root alongside source code. This is a deliberate design: the knowledge base lives *with* the project, not separate from it, enabling agents to treat project knowledge as co-located context.

`.napkin/` is an Obsidian vault that napkin creates, operates on, and keeps compatible -- but never requires Obsidian to be installed. It is a format choice, not a dependency. Wiki-style links, frontmatter, folder structure, and the entire Obsidian ecosystem are available, but the system works standalone. Napkin also renders the vault's link graph in a native window via Glimpse -- agents first, but humans deserve a view too.

Templates (`coding`, `company`, `product`, `personal`, `research`) scaffold domain-specific directory structures, starter notes, Obsidian templates, and a `NAPKIN.md` skeleton. Each template defines not just folders but what the overview index extracts -- the overview is template-aware. A coding project template creates `decisions/`, `architecture/`, `guides/`, `changelog/` with structured frontmatter schemas like `status: proposed` on decisions.

## Core Mechanism

### Progressive Disclosure (L0-L3)

The central innovation is a four-level disclosure model that respects context window budgets. Each level's output includes hints that teach the agent how to go deeper: overview hints toward search, search hints toward read -- a progressive disclosure of the workflow itself.

**Level 0 -- NAPKIN.md (~200-500 tokens):** A pinned context note read on every session. Functions like CLAUDE.md but for the knowledge base. Contains project goals, conventions, key decisions. This is the "always loaded" layer that orients the agent to the knowledge base's purpose and structure.

**Level 1 -- Overview (~1-2K tokens):** Generated by `getOverview()` in `src/core/overview.ts`. This is the compressed keyword map that replaces vector search for navigation. The algorithm:

1. Lists all `.md` files and groups them by directory (respecting a configurable `maxDepth`).
2. For each folder, builds a term-frequency map from weighted sources: headings get 3x weight, filenames and frontmatter titles get 2x, body text gets 1x.
3. Extracts both unigrams and bigrams, filtering stopwords.
4. Computes TF-IDF scores across folders (each folder is a "document" for IDF purposes).
5. Selects top-K keywords per folder, with bigram suppression -- if a bigram is selected, its constituent unigrams are suppressed to avoid redundancy.
6. Collects inline tags and frontmatter tags per folder.

The output gives agents a topical map without any LLM calls:
```
architecture/
  keywords: overview, dependencies, design
  notes: 9
```
This is enough to decide whether to search deeper. The weighting scheme (headings 3x, filenames 2x, body 1x) means the overview reflects what each folder is *about*, not just what words appear most frequently.

**Level 2 -- Search (~2-5K tokens):** The `searchVault()` function in `src/core/search.ts` uses MiniSearch (a JavaScript BM25 implementation) with a composite three-signal ranking:

```
composite = BM25_score + backlink_count * 0.5 + recency_normalized * 1.0
```

The three signals:
- **BM25** -- Standard term frequency / inverse document frequency. Searches both `basename` (2x boost) and `content` fields. Enables fuzzy matching (0.2) and prefix matching.
- **Backlinks** -- The system crawls all `[[wikilink]]` references across the vault and counts inbound links per file. Files with more incoming links rank higher -- effectively a simplified PageRank using Obsidian's link graph.
- **Recency** -- File modification time (`mtime`), normalized to [0,1] range, with 1.0 weight as a tiebreaker.

Search results include contextual snippets: for each matching line, the surrounding N lines (configurable, default 3) are included. Snippets merge overlapping ranges to avoid redundancy. Returns match-only lines by default with no surrounding context, to avoid overwhelming the model with information.

**Critical design choice: scores are hidden from the output.** If an agent sees numerical scores, it anchors on the numbers instead of judging relevance from snippets. The score drives ordering, but the agent never sees it. This is designing for model psychology, not human psychology.

**Level 3 -- Read (full file):** Direct file content, already implemented via `napkin read`.

### Search Index Caching

The `src/utils/search-cache.ts` implements a fingerprint-based cache invalidation strategy:

1. Computes an MD5 fingerprint from all `.md` file paths concatenated with their `mtimeMs` values.
2. If the fingerprint matches the cached version, loads the serialized MiniSearch index from disk (JSON).
3. If fingerprint misses, rebuilds the full index and persists it.

The cache stores serialized MiniSearch JSON, document metadata (without content -- content is re-read for snippets), and the backlink count map. The first search in a session pays the full indexing cost, but subsequent searches are near-instant.

### File Resolution

Files can be referenced two ways: by name (wikilink-style, searches all `.md` files by basename) or by exact path from vault root. This mirrors how Obsidian itself resolves links, making the vault fully Obsidian-compatible while also agent-friendly.

### Auto-Distillation System

The distillation system is napkin's write-back mechanism -- how the vault accumulates knowledge over time without manual note-taking. Background extraction runs on timers (or via `/distill` command):

1. The distill extension receives the full conversation the user is having with the agent
2. A dedicated prompt explores the existing vault content to understand what knowledge already exists
3. The system identifies knowledge gaps -- new information from the conversation that is not yet captured
4. New notes are created linking to relevant existing notes using wikilinks
5. Template schemas are applied for consistency (frontmatter structure matches the vault's template)
6. Knowledge is captured as a side effect of normal conversation, not a separate action

This creates a read/write loop: the agent reads from the vault at session start (napkin-context), works during the session, and writes back at session end (napkin-distill). Over time, the vault accumulates project knowledge automatically. The distill process does not just dump notes into a folder -- it connects new knowledge to what is already there, maintaining the link graph that powers the backlink signal in search.

## Design Tradeoffs

### BM25 vs. Embeddings: The Core Bet

Napkin makes a strong, opinionated bet: **no vectors, no embeddings, no graph construction**. The design document states explicitly: "BM25 + link graph + recency > embeddings for structured knowledge bases." The benchmark results validate this claim.

The reasoning is multi-layered:

1. **Zero preprocessing cost** -- No embedding model to run, no vector store to maintain, no chunking strategy to tune. Files are the index. When a file changes, the BM25 index updates immediately via fingerprint invalidation. Embedding pipelines have lag.
2. **Interpretability** -- BM25 matches are explainable (these terms matched). Embedding similarity is a black box.
3. **No infrastructure** -- No vector database (Chroma, Pinecone, Qdrant) to deploy, configure, or pay for. The entire system runs locally with zero external dependencies.
4. **Token efficiency** -- Progressive disclosure means the agent pays only for what it reads. A vector search returns K chunks regardless of relevance.
5. **The right model makes the decision** -- Instead of a small embedding model making retrieval decisions via cosine similarity, the full-capability LLM navigates the knowledge base using its reasoning ability. The overview provides the map; the agent provides the navigation intelligence.

The tradeoff is real: BM25 cannot match semantic similarity across vocabulary gaps ("authentication" won't match "login" unless both appear in documents). The backlink signal partially compensates -- if both terms appear in linked notes, the graph connectivity surfaces them -- but this is weaker than embedding cosine similarity for synonym matching.

### Comparison with memsearch

Zilliz's memsearch project (extracted from OpenClaw's memory system) takes a different approach to the same problem. Memsearch uses hybrid search: BM25 for keyword matching AND vector embeddings for semantic recall, with RRF (Reciprocal Rank Fusion) reranking. It also uses SHA-256 content hashing to avoid re-embedding unchanged content and a file watcher for live sync.

The fundamental architectural difference: memsearch treats vector databases as derived indexes (rebuildable from markdown source of truth), while napkin eliminates vectors entirely. Memsearch acknowledges BM25's vocabulary gap weakness and compensates with embeddings; napkin accepts the gap as an acceptable tradeoff for zero-infrastructure simplicity.

Both share the markdown-first philosophy: markdown files are the source of truth, and any indexes (BM25, vector, or both) are derived artifacts. This contrasts with database-first memory systems where the database is primary storage and markdown export is secondary.

### Per-Round Notes vs. Full Sessions (Benchmark Design)

The LongMemEval benchmark reveals a crucial design choice. Chat history is split into **per-round notes** (~2.5K chars each) rather than full sessions (~15K chars). This gives BM25 better granularity -- a full session dilutes term frequency across many topics, while a per-round note concentrates terms for the specific exchange. Notes are organized in **day directories** so the overview's TF-IDF keywords provide temporal topic navigation.

This is not just a benchmark trick -- it is a general principle for agent memory: smaller, more focused notes outperform large aggregated documents for keyword-based retrieval. The implication is architectural: memory systems should write many small files rather than appending to large session logs.

### Obsidian Compatibility vs. Pure Agent Design

Napkin maintains full Obsidian compatibility (auto-generates `.obsidian/` config, supports wikilinks, tags, frontmatter, canvas, bookmarks). This means a human can open the same vault in Obsidian while an agent works with it via the CLI/SDK. The cost is complexity: the codebase must understand Obsidian's config format, template system, and link resolution rules. The benefit is that the vault is not locked into agent-only usage -- it remains human-readable and human-editable.

### Score Hiding as Model Psychology

The decision to hide search scores from agent output is a deliberately psychological design. Visible numeric scores cause anchoring bias in LLMs -- the model fixates on the number rather than evaluating content quality. By presenting only ordered results with content snippets, the agent makes relevance judgments from semantic evaluation of the actual text. Scores drive ordering invisibly.

This principle extends beyond napkin: any system that provides retrieval results to an LLM should consider whether exposing confidence scores helps or hinders the model's ability to judge relevance.

## Failure Modes & Limitations

1. **Vocabulary mismatch** -- BM25 fundamentally requires term overlap. If notes use "authn/authz" but queries use "authentication/authorization", matches will fail. No semantic bridging exists. This is the primary weakness compared to embedding-based systems.

2. **Scale ceiling** -- The full-file MiniSearch index is rebuilt when the fingerprint changes. For very large vaults (10K+ files), this could become slow. The cache helps for repeated searches within a session, but first-search latency grows linearly with vault size.

3. **Backlink signal requires discipline** -- The PageRank-like boost only works if notes actually link to each other. A vault of unlinked notes gets no graph signal, degrading to pure BM25. This creates a bootstrapping problem: new vaults have no links. The auto-distillation system partially addresses this by creating links during knowledge capture.

4. **No cross-vault search** -- Each vault is independent. There is no mechanism to search across multiple projects, which limits utility for agents working across repositories.

5. **Snippet extraction is naive** -- Snippets are extracted by simple term matching on lines. No sentence boundary detection, no semantic relevance filtering. Long matching lines may produce noisy snippets.

6. **Overview depth limitation** -- The TF-IDF overview only processes folders up to `maxDepth`. Deeply nested directory structures lose keyword visibility, which could cause agents to miss relevant subfolders.

7. **Abstention weakness** -- The LongMemEval benchmark shows 50% accuracy on abstention tasks (knowing when *not* to answer). BM25 always returns results ranked by score; there is no calibrated confidence threshold to signal "I don't know." This is a significant gap for production memory systems where false confidence is worse than admitting ignorance.

8. **Distillation quality dependence** -- The auto-distillation system's output depends on the LLM's ability to identify and extract relevant knowledge from conversations. Poor distillation (irrelevant notes, missed connections, wrong frontmatter schemas) degrades vault quality over time. There is no quality validation for distilled notes.

9. **Temporal reasoning limitations** -- While day-directory organization enables temporal keyword extraction in the overview, the system has no explicit temporal index. Queries like "what did we discuss last Tuesday" require the agent to navigate date folders manually.

## Integration Patterns

### Agent Memory via Pi Extensions

Napkin was built for `pi` (pi-coding-agent), a multi-platform agent framework, leveraging its extension system. Two extensions create the read/write memory loop:

- **napkin-context** -- Injects vault overview (L0 + L1) into the agent's system prompt on session start. This is the automatic "orient the agent" step. The overview loads automatically at the beginning of every conversation.
- **napkin-distill** -- Forks the current session and spawns a sub-agent to distill knowledge from the conversation into the vault. Runs in background as a timer in a pi extension, non-blocking. The full conversation feeds a dedicated prompt that explores existing knowledge, finds relevant notes, and links new information to them.

This creates the accumulation loop: read at session start, work during session, write back at session end. Over time, the vault becomes a persistent project memory that improves with every agent interaction.

### SDK for Programmatic Access

The SDK API (`new Napkin(path)`) enables programmatic integration without CLI overhead. All methods return typed data and throw on failure -- no stdout parsing needed. This makes napkin embeddable in larger agent architectures as a memory subsystem.

```typescript
const napkin = new Napkin("./my-project");
const overview = await napkin.overview();      // L1: keyword map
const results = await napkin.search("auth");   // L2: BM25 + backlinks
const content = await napkin.read("auth.md");  // L3: full file
```

### Template System for Domain Adaptation

The 5 built-in templates demonstrate how the same progressive-disclosure mechanism adapts to different domains:

- **coding**: `decisions/`, `architecture/`, `guides/`, `changelog/` with frontmatter schemas (status: proposed, decided, superseded)
- **company**: Organizational knowledge, processes, team structures
- **product**: Product specifications, user research, roadmap
- **personal**: Personal knowledge management, notes, journal
- **research**: Literature notes, experiment logs, findings

Each template defines directory structure, starter notes (pre-populated `_about.md` files), note templates (frontmatter schemas), and NAPKIN.md skeleton. Custom templates are supported via the same interface, enabling teams to standardize vault structure.

### Graph Visualization

Napkin renders the vault's link graph via Glimpse in a native window. This gives both agents and humans a visual representation of the knowledge base's connectivity. The graph visualization surfaces:
- Highly-connected hub notes (central concepts)
- Disconnected clusters (knowledge silos)
- Orphan notes (potential linking opportunities)

## Benchmarks & Performance

### LongMemEval (ICLR 2025)

The headline benchmark tests long-term conversational memory across 500 questions:

| Dataset | Sessions/Q | pi + napkin | Best Prior System | GPT-4o Full Context |
|---------|-----------|-------------|-------------------|---------------------|
| Oracle | 1-6 | 92.0% | 92.4% (GPT-4o+CoN) | 92.4% |
| S | ~40 | **91.0%** | 86% (Emergence AI) | 64% |
| M | ~500 | **83.0%** | 72% (GPT-4o RAG) | n/a |

The S-dataset result is the most significant: 91% vs 86% for the best prior system, and vs 64% for GPT-4o with full context. This means BM25 with progressive disclosure outperforms both embedding-based RAG and context-window stuffing. The M-dataset (1.5M tokens per question) at 83% vs 72% demonstrates that the approach scales -- context-window stuffing fails at this scale, but progressive disclosure with BM25 still works.

Per-ability breakdown on S-dataset reveals strengths and weaknesses:
- Information Extraction: 97.1% (nearly perfect)
- Knowledge Updates: 100% (BM25 + recency handles this well)
- Temporal Reasoning: 90.3% (day-directory organization helps)
- Multi-Session Reasoning: 88.9% (link graph assists cross-reference)
- Abstention: 50.0% (significant weakness -- no confidence calibration)

**Context on the leaderboard**: Since napkin's benchmark, JordanMcCann's agentmemory system achieved 96.2% on LongMemEval using a six-signal hybrid approach (semantic similarity + BM25 + graph spreading activation + node importance + calibrated confidence + temporal proximity). This demonstrates that adding semantic signals on top of BM25 can further improve results, but napkin's BM25-only approach remains competitive at a fraction of the complexity.

### Key Benchmark Design Decisions

1. **Per-round notes** -- Each round (~1 user message + assistant responses) becomes one note (~2.5K chars). This is more granular than session-level notes (~15K chars), giving BM25 better term concentration.
2. **Day directories** -- Notes organized by date enable temporal keyword extraction in the overview.
3. **Full assistant responses included** -- Many questions ask about prior assistant statements. Truncating assistant content drops answer material.
4. **Scenario date in system prompt** -- Prevents the model from using real current date for relative time calculations.

### Additional Benchmarks

- **HotpotQA** -- Multi-hop QA with 10 context paragraphs (2 gold, 8 distractors). Tests cross-note information chaining.
- **LoCoMo** -- 10 conversations, 699 questions across single-hop, multi-hop, and temporal reasoning.

### Runtime Performance

No published latency numbers, but the architecture implies:
- First search: O(N) where N = number of files (builds MiniSearch index)
- Subsequent searches: O(K) where K = result count (cached index, load from JSON)
- Overview generation: O(N) always (reads all files for TF-IDF)
- Distillation: O(conversation_length) for LLM processing

The search cache is the critical optimization -- without it, every search would rebuild the index.

## Comparative Analysis

**vs. RAGFlow**: Opposite ends of the spectrum. RAGFlow is a Docker-based infrastructure system requiring MySQL, Elasticsearch, Redis, and MinIO (16GB+ RAM). Napkin is a zero-dependency local library. RAGFlow provides multi-recall hybrid retrieval (BM25 + vectors + GraphRAG + RAPTOR); napkin provides BM25 + backlinks + recency only. RAGFlow is designed for production multi-tenant deployment; napkin is designed for single-agent project memory. The philosophical difference: RAGFlow invests in preprocessing (parsing, chunking, embedding, graph construction) to improve retrieval quality; napkin invests in zero preprocessing to maximize freshness and simplicity.

**vs. PageIndex**: Both reject embedding-based retrieval, but through different mechanisms. PageIndex uses LLM reasoning over a tree structure (the agent navigates); napkin uses BM25 keyword matching (the index retrieves). PageIndex is designed for static document Q&A (financial reports, legal filings); napkin is designed for evolving project memory (conversations, decisions, architecture notes). PageIndex requires LLM API calls for both indexing and retrieval; napkin requires zero LLM calls for search (only for distillation).

**vs. memsearch**: Zilliz's memsearch uses hybrid BM25 + vector search with RRF reranking, maintaining markdown as source of truth but adding embedding-based semantic recall. Memsearch acknowledges BM25's vocabulary gap and compensates with vectors; napkin accepts the gap. Memsearch requires an embedding model and vector database; napkin requires nothing. The tradeoff: memsearch handles synonym matching better; napkin has zero infrastructure cost.

**vs. Claude-Mem/ClawMem**: Other agent memory systems inspired by similar principles (markdown-first, progressive disclosure) but with different implementations. These typically add vector search or hook-based memory capture. Napkin's unique contribution is proving BM25-only is sufficient for benchmark-leading performance.

## Implications for Meta-KB

Napkin's success with BM25 on markdown challenges the assumption that embedding-based RAG is necessary for knowledge base retrieval. Key lessons for meta-kb:

1. **Progressive disclosure directly applicable** -- Instead of dumping the full wiki into context, a tiered approach (overview > search > read) would respect context window budgets. Meta-kb's wiki compilation could benefit from generating an L1 overview (keyword map by topic area) that agents consult before diving into specific articles.

2. **TF-IDF keyword extraction for navigation** -- Converts a directory structure into a navigable topic map without any LLM calls. Meta-kb could generate an automatic overview of its `raw/` and `wiki/` directories using the same weighted TF-IDF approach (headings 3x, filenames 2x, body 1x).

3. **Backlink-weighted search** -- Structure (wiki links) can substitute for embeddings when the knowledge base is well-connected. Meta-kb's wiki cross-references between topics could power a backlink signal for retrieval, surfacing highly-connected concepts.

4. **Per-note granularity** -- Smaller, more focused documents outperform large aggregated ones for keyword retrieval. This has implications for how meta-kb structures its `raw/` files -- the current one-file-per-source approach is well-suited to BM25 retrieval.

5. **Score hiding for LLM consumption** -- If meta-kb ever provides retrieval results to an LLM (e.g., for compilation), hiding numeric scores would prevent anchoring bias. Present ordered results with content snippets, not confidence numbers.

6. **Auto-distillation pattern** -- The read/write loop (inject context at start, distill knowledge at end) is a generalizable pattern for any knowledge base that interacts with LLM agents. Meta-kb could implement a similar loop where wiki compilation sessions produce both wiki content and metadata updates to source files.

7. **Zero-infrastructure viability** -- Napkin's benchmark results prove that local-first, file-based knowledge systems can outperform infrastructure-heavy alternatives. This validates meta-kb's file-based architecture against the temptation to add vector databases or graph stores.
