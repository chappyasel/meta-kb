# OpenViking

> Context database for AI agents that uses a filesystem paradigm to unify memory, resources, and skills management with L0/L1/L2 tiered loading and automatic session compression -- built by Volcengine (ByteDance).

## What It Does

OpenViking replaces fragmented vector stores with a unified virtual filesystem (`viking://` protocol) where all agent context -- memories, resources, skills -- lives as structured entries with unique URIs. Agents interact with context like developers interact with files: `ls`, `find`, `tree`, `grep`. The system automatically processes content into three tiers on write: L0 (one-sentence abstract, ~100 tokens), L1 (overview with core information, ~2k tokens), and L2 (full content). Retrieval uses directory recursive search: intent analysis generates multiple retrieval conditions, vector search locates high-score directories, and secondary retrieval refines within directories recursively. At session end, conversations are automatically compressed into long-term memory.

## Architecture

Python server (`pip install openviking`) with optional Rust CLI. The core is a virtual filesystem mapping everything under `viking://resources/`, `viking://user/`, and `viking://agent/` paths. Each entry gets L0/L1/L2 layers generated by a VLM model. Embedding model handles vectorization for semantic retrieval within the directory structure. Requires Go 1.22+ for AGFS components and a C++ compiler for core extensions. Supports multiple LLM providers via LiteLLM and multiple embedding providers (Volcengine, OpenAI, Jina, Voyage, Gemini). VikingBot provides an AI agent framework on top. OpenClaw plugin integration shows 49% improvement over baseline OpenClaw with 83% reduction in input token cost on LoCoMo.

## Strengths

- Filesystem metaphor makes context management observable and debuggable -- every retrieval has a visible trajectory through directories, unlike black-box vector search
- L0/L1/L2 tiering is a principled solution to token bloat: agents read abstracts first, drill down only when needed
- Unified management of memory, resources, and skills in one system eliminates the fragmentation problem where each lives in a different store
- Automatic session compression creates a genuine self-improvement loop -- agents get smarter across sessions without manual curation

## Limitations

- Heavy infrastructure requirements: Python 3.10+, Go 1.22+, C++ compiler, plus LLM and embedding API keys
- Volcengine-first design means some documentation and community discussion is Chinese-language
- AGPL-3.0 license for the main project is restrictive for commercial use
- L0/L1/L2 layer generation requires VLM calls for every ingested resource, adding cost at write time
